# Technical Design

_Living document — edited in place, history lives in git. One section per design area; each records the current design, its rationale, **Rejected:** alternatives with why, and **Revisit when:** triggers, so settled questions aren't re-litigated._

## 1. Architecture overview

Gmail filters funnel all job-alert email under one label; a Google Apps Script collector picks it up on a time trigger (cadence: §7), cleans it offline, and stores clean text in Airtable (RawEmails queue + Vacancies decisions); a scheduled Claude (Cowork) run screens the queue against versioned instructions, writes decisions, and reports to Ivan. The flow diagram and component table live in the [README](../README.md) — not duplicated here.

## 2. Collector — fetch & write pipeline

- **Sub-batch pipeline with a forward-progress guarantee.** The run is an interleaved fetch → upsert → label loop over sub-batches (`CONFIG.SUB_BATCH_SIZE = 5`), each committed before the next — not a two-phase fetch-all-then-write-all. The runtime budget (`MAX_RUNTIME_MS`, against the ~6-minute Apps Script execution limit) is checked once per sub-batch, and the first sub-batch always runs: every run commits at least one sub-batch, and a timeout or crash loses at most one. **Rejected:** separate budget breaks in a two-phase fetch/write structure — under sustained slow-fetch latency the write-phase guard tripped before writing anything, re-collecting the same head-of-queue messages with no forward progress. Guarded by the forward-progress test in `tests/collect-loop.test.js`.
- **Write-then-label ordering.** The Airtable write happens before the Gmail label; a crash in between re-presents the message next run. No data loss by design — the dedupe below makes the retry safe.
- **Dedupe on retry.** Airtable upsert (`performUpsert` on `MessageId`) rather than pre-checking existing rows. **Revisit when:** rows become per-vacancy instead of per-email — switch the dedupe key to `gmailMessageId + urlHash`.
- **Read-side: transient-retry then poison isolation — symmetric with the write side.** The per-message Gmail read mirrors the write side's transient-vs-deterministic split, in **two narrow `try/catch` stages** with separate failure handling. (1) `Gmail.Users.Messages.get` runs through `gmailReadWithRetry_`, which retries a **transient** read blip (`[1s, 2s, 4s]`, budget-aware) and, if it persists, leaves the message **uncollected** (unlabelled → re-presents next run, idempotent on the `MessageId` upsert) and **never** `failed` — a transient read is not poison. A persistent read fails **loud** (run ends Failed) so a Gmail-read outage can't silently stall the queue. (2) A deterministic **parse** failure (`headerMap_`/`processMessage_` — a malformed MIME tree) still labels the message `failed` (quarantine + MIME-tree forensics log), exactly as before, and does **not** trip the read canary (a quarantined poison is *handled*, not an un-collected read). Splitting the formerly-**single** `try/catch` into two is the fix for a **latent data-loss bug**: the old catch labelled **any** throw — including a transient `get` `429`/`5xx`/network blip — `failed`, evicting a perfectly good message from `CONFIG.QUERY` *permanently*. The once-per-run `Gmail.Users.Messages.list` is wrapped the same way (a transient blip heals; a persistent failure propagates → run Failed), removing the spurious single-blip run failure. A run hitting any combination of **read failure**, **upsert failure**, and **footer-marker miss** throws **one** error naming *every* present signal (precedence: reads, then writes, then the footer alarm), so no co-occurring signal is swallowed (F1, PR #17). The read-failure alarm also fires in **`DRY_RUN`** — DRY_RUN does real reads, so a persistent read outage must fail loud there too or dry-run validation silently misses it (Codex F1[P2], PR #25); the write-path signals do **not** (an upsert never runs in DRY_RUN, and a footer miss is a side-effect-only template alarm deliberately suppressed in dry runs). **Out of scope (Option B):** parsing the HTTP status out of the thrown error to skip a genuine `404` — it depends on the Gmail advanced service's runtime error shape (`GoogleJsonResponseException`), which the stubbed Node harness can't verify ("a stubbed runtime hides runtime-only behavior"), so the read retries on **any** throw; a `404`-skip is a possible future follow-up gated on a real-runtime probe, only if alert noise appears (owner decision 2026-06-15). **Rejected:** merging the two retry wrappers — different transport contracts (Gmail throws on non-2xx and returns no response; Airtable uses `muteHttpExceptions` + `getResponseCode()`); they share only the backoff constant and the budget predicate.
- **Write-side per-record isolation — any failed sub-batch (poison or transient).** A single record-specific failure fails the whole all-or-nothing sub-batch PATCH — a deterministic `4xx` reject (e.g. `422` validation) **or** a `5xx`/transport blip that trips on one record — so without isolation the good siblings are never `collected` and are re-fetched on *every* run (each ending Failed) until the sub-batch composition happens to change. `airtableUpsert_` returns the numeric HTTP code; `attemptUpsert_` classifies it `ok` (`200`) / `transient` (`429`/`5xx`/network throw, `isTransientWriteFailure_`) / `poison` (any other `4xx`). **Any non-`ok` batch** is re-sent record-by-record on its own PATCH (through `airtableUpsert_`, so the transient-retry wrapper below composes with the isolation), and each record takes its per-record outcome: **`200`** → `collected`, unblocking a healthy sibling instead of stranding it behind one bad record; **`transient`** → left uncollected, retried next run, **never** `failed` on a single failure (a transient is not poison; a *record-specific* one that persists is capped across runs — see the next bullet); **`poison`** (`4xx`) → `failed` **only if ≥1 sibling upserted `200`** that run (the **quarantine guard** — proof the endpoint/auth/schema is healthy and the reject is record-specific). Removing the old `transient` early-`continue` — which left a whole transient sub-batch uncollected on every run — is what unblocks the healthy siblings: a record-specific `5xx` no longer holds them hostage. **Systemic failure unchanged:** if **zero** records succeed — every record `4xx` (a systemic `401`/`404`/`422 UNKNOWN_FIELD` from bad auth, wrong endpoint, or schema drift) **or** an all-records transient outage (no healthy sibling) — it quarantines **nothing for a fresh (never-struck) record**, leaves the whole sub-batch uncollected, and fails loud, so a deploy mistake or an outage can never *mass*-quarantine the queue (`failed` excludes a message from `CONFIG.QUERY` permanently). The one bounded exception is the **struck-then-outage residual** — a *transient* record that already earned record-specific strikes can finish capping to `failed` even in such a run (sticky record-specificity; see the cap bullet below) — never a fresh record, so never a mass quarantine. `upsertFailures` counts **once per sub-batch** that leaves anything stuck (not per record — F-P3), and the run ends Failed so the GAS failure email fires. **Accepted cost:** during a true systemic outage every failed sub-batch now individually re-sends (≤ `SUB_BATCH_SIZE` records × the retry schedule), bounded by `MAX_RUNTIME_MS` (the budget predicate stops the run before it burns the execution limit) — the same cost profile the poison path already had; an outage isolation fast-path is a possible future optimisation, deliberately not added here. **Accepted limitation:** a poison record *alone* in its sub-batch has no sibling to prove record-specificity, so it stays uncollected and keeps alerting until it re-batches with others or a human handles it — conservative by design. **Foundation for the repeatedly-transient cap:** the per-record `transient` outcome this exposes — a record-specific write that is stuck but not poison — is where the **repeatedly-transient write cap** (next bullet) hangs its cross-run strike counter. **Rejected (still standing for the *deterministic* case):** a cross-run per-message failure counter that would `failed` a *deterministic* reject after N failed runs — persistent per-message state, it does **not** unblock the good siblings in the same PATCH, and it is slower to quarantine than the within-run individual isolation above, which is record-specific and immediate (owner decision 2026-06-12). That same counter **does** fit the **repeatedly-transient** case — a stuck transient record has no within-run remedy, since retrying it only fails again — so it is **implemented** there (next bullet), built on the isolation prerequisite this bullet landed.
- **Repeatedly-transient write cap — `failed` after N record-specific strikes.** A record whose *own* PATCH trips a `transient` outcome on every run is otherwise retried **forever** and fails the run **every** run (the unbounded `upsertFailures` alarm storm the backlog flagged): the per-record isolation above unblocks its healthy *siblings* but never caps the stuck record itself. So each **record-specific** transient strikes a **cross-run counter**, gated **`anyHealthy || priorStrikes > 0`**: the **first** strike needs a same-run healthy sibling (proof the system is up and the failure is this record's), but once a message carries a strike the counter is **sticky** — its subsequent **solo** failures keep striking with no fresh sibling required. Stickiness is load-bearing: #26 collects the healthy siblings out of `CONFIG.QUERY`, so a genuinely-stuck record is **alone** on every later run, and a per-run-only health proof would freeze its counter at the run-1 value so the cap is never reached (Codex F1, PR #27). At `MAX_TRANSIENT_WRITE_RETRIES` (`CONFIG` default **5** ≈ 2.5 h of sustained record-specific failure at the ~30-min cadence; runtime-tunable via `getIntProp_`, bounds `[1, 100]`, like `MAX_MESSAGES`) **consecutive** strikes the record is `failed` (reusing `FAILED_LABEL_NAME` — no new label) so it stops re-presenting, with one final quarantine notification. **The counter lives in Script Properties, one integer per message keyed `wretry:<messageId>`** — it cannot live in Airtable, because the failing write means the message has no row (owner decision 2026-06-15). It is **loaded once per run** into an in-memory map (so clear-on-success doesn't fire a `deleteProperty` for every collected message — only the rare struck-then-recovered one), mutated in memory, and persisted immediately (`setProperty` on a strike, `deleteProperty` on clear/quarantine). **Strike semantics are consecutive:** any successful upsert of that message clears its counter (the `ok` collect paths, whole-batch **and** per-record); a systemic-outage run is **neutral** (neither strikes nor clears). **The mass-quarantine guard holds for the bulk of the queue:** a **fresh** (never-struck) message in a systemic outage **never** strikes (no first-strike sibling proof), so a multi-hour Airtable outage cannot quarantine fresh traffic. **Two accepted residuals** (owner-ratified 2026-06-15, resolving Codex F1 — Option 1 "sticky record-specificity"): (1) **struck-then-outage** — a record that already earned ≥1 genuine record-specific strike and then faces a *sustained* systemic outage while solo can accrue to the cap and be quarantined; bounded to already-suspicious records (**never** fresh ones, so never a *mass* quarantine — at most the handful currently near N), and recoverable (remove the `failed` label to retry); the quarantine notice + the OPERATIONS triage make it observable. (2) **solo-from-start never caps** — a record alone in the queue for its entire failing lifetime never earns a first strike, so it is never capped (same as the prior unbounded behaviour for that ultra-rare case; requiring first-strike sibling-proof is the deliberate price of never quarantining a fresh message during an outage). A quarantined record is **handled** (like the poison quarantine), so it does **not** count toward `upsertFailures`; it raises its **own** fail-loud signal instead — the end-of-run throw names every present signal in precedence **reads → upserts → quarantines → footer** (F1, none swallowed). **`DRY_RUN` does no strike, no quarantine, no Script Properties write/delete** — the whole upsert/isolation block is short-circuited before any of it (the map may be *loaded*, a read, but is never mutated); stated and tested explicitly rather than inherited (Codex F1[P2], PR #25). The `setProperty`/`deleteProperty` calls sit **outside** any upsert `try`, so a Script Properties error surfaces with its own stack rather than being masked as a transient write failure (narrow-catch, F-P2). **No in-flight `processing` marker is set** (the original Make scenario never set one either): single-flight + the idempotent write-then-label already give crash-safety, so a mid-flight marker would add persistent state for no benefit. The never-set processing token that `CONFIG.QUERY` used to exclude was dropped in the label-rename slice (previously left as a harmless no-op). **Accepted cost:** a message manually deleted from Gmail while struck orphans a few-byte `wretry:` key — rare, bounded by Script Properties' generous cap; a GC sweep is a possible future follow-up, not this slice. **Rejected:** a distinct quarantine label (`stuck`) — reuse `failed`; time-based (rather than run-count) strike semantics. **Revisit when:** the quarantine alarm proves too noisy, or orphaned `wretry:` keys accumulate enough to warrant a sweep.
- **Transient-retry/backoff wrappers — heal a blip within the run.** Two sibling wrappers share the backoff constant (`CONFIG.RETRY_BACKOFF_MS`; `[1s, 2s, 4s]` = 3 retries, 4 attempts) and the budget predicate, but stay **separate** for their different transport contracts: `gmailReadWithRetry_` for the idempotent Gmail **reads** (above — retries on *any* throw, no status parsing) and `airtableFetchWithRetry_` for the Airtable **writes** (this bullet). Every Airtable `UrlFetchApp.fetch` goes through `airtableFetchWithRetry_`, which retries a **transient** outcome — HTTP `429`, any `5xx` (`isTransientWriteFailure_`), or a thrown transport exception — with `[1s, 2s, 4s]` backoff (`CONFIG.RETRY_BACKOFF_MS`; 3 retries, 4 attempts). A `200` or a **deterministic 4xx** (`400`/`401`/`404`/`422`) passes straight through (retrying a validation/auth reject only burns budget). The wrapper sits **below** the per-record isolation contract above: callers see the *final* code after retries, so a transient that persists still classifies `transient` — its record left for next run, fail-loud, via the isolation above — and the quarantine guard is unchanged. The **idempotent `MessageId` upsert** makes a retried PATCH safe — no duplicate rows. **Budget-aware:** on the collector path the wrapper is threaded a predicate `(ms) => isOverRuntimeBudget_(startMs, now()+ms)` and **skips a backoff sleep that would cross `MAX_RUNTIME_MS`**, returning the last code rather than risk a hard ~6-minute kill mid-write — the forward-progress guarantee and the per-sub-batch budget check are preserved. The sleep fn, clock, and backoff schedule are injectable so unit tests verify the sequence without real sleeps (`tests/reliability-helpers.test.js`); recovery-composes-with-isolation is pinned in `tests/collect-loop.test.js`. **`DELETE` is the exception — `retryOnThrow:false`:** re-deleting an already-gone id returns `404 MODEL_ID_NOT_FOUND` (verified against the Airtable API), so a DELETE retried after a *transport throw* (where a prior success can't be ruled out) could 404 a delete that actually landed and fail the run spuriously. So deletes retry on **`429`/`5xx` only** (a server-side reject removed nothing — safe), and a transport throw propagates on the first attempt (pre-wrapper behaviour). The purge path gets **no budget predicate** — its per-run delete volume is small (nightly delta), so the worst-case total backoff stays short; and a purge is non-critical and self-correcting if it ever did hit the limit (committed deletes are durable; the next night re-counts). **Rejected:** retrying deletes on a transport throw (the `404`-on-phantom-success risk); making the backoff schedule a runtime Script Property (a `CONFIG` constant is enough — tune and redeploy). **Revisit when:** a large post-M6.2 catch-up purge routinely deletes hundreds of rows per run — then add a budget predicate to the purge path too (owner/architect decision 2026-06-12).
- **Single-flight.** A `LockService` guard (`tryLock(0)`, exit cleanly if held, release in `finally`) prevents overlapping scheduled runs — duplicate writes and label races.
- **Runtime controls, not redeploys.** `MAX_MESSAGES` (fetch cap; `0` = pause switch), `MAX_TRANSIENT_WRITE_RETRIES` (the repeatedly-transient quarantine cap, `[1,100]`; above), and `DRY_RUN` (log would-be writes/labels, touch nothing) are Script Properties read at runtime. Semantics and procedure: [OPERATIONS — Collector: routine procedures](OPERATIONS.md#collector-routine-procedures).

## 3. Collector — offline link cleanup

Job-alert emails wrap links in tracking redirectors, which bloats `CleanText` (measured: ~10 KB of a 19.4 KB body was tracking-URL base64) and hides real job URLs from screening.

**Decision: the collector cleans links entirely offline — it makes no network calls of any kind** (no `UrlFetchApp`, no fetching/following/probing). Before `CLEAN_REGEX`, for every URL in the body (both `href="…"` values and bare-text URLs) it (a) decodes an embedded destination and (b) strips `utm_*` analytics params, swapping the cleaned form in place. Operational description and the per-run `Links: decoded=N utm_stripped=M bytes_saved=B` log line (the only output — no Airtable schema field): [OPERATIONS — Collector: offline link cleanup](OPERATIONS.md#collector-offline-link-cleanup).

- **Value-guard only — no param-name allow-list, no tracker-host list.** The embedded destination is the **first** query param in document order whose URL-decoded value is itself an absolute `http(s)` URL or an absolute path (`/…`, prepending the tracker's scheme+host). **Rejected:** a curated list of redirect param names (`url`, `u`, `dest`, `redirect_uri`, …) — a maintenance burden that silently misses any name not enumerated (`redirect_url`, `dest_url`, `r2`, …); the "value must be a URL/path" guard is zero-maintenance and future-proof. **Accepted cost:** if a tracker carries a non-destination URL-valued param *before* the real one in document order (e.g. `?img=https://cdn/logo.png&url=…`), the wrong value is picked — rare in click-trackers, deliberate trade-off (decided with Ivan).
- **`HtmlLength` parity.** `HtmlLength` stays the **original** body length (parity with the original Make scenario's `length(1.htmlBody)`); only `CleanText`/`CleanLength` reflect the cleanup. With neither an embedded destination nor a `utm_` param present, the transform is a **byte-identical no-op**.
- **Rejected: network redirect resolution** (`UrlFetchApp`, hop on `3xx` `Location`, cap per run — prototyped in PR #5, closed unmerged). Judged both risky and incomplete: (1) **side-effect risk** — tracker links are not all idempotent GETs; some are one-click unsubscribe or 1-click-apply endpoints, and even a `HEAD`/no-body hop can trip them; (2) **incompleteness** — opaque tracker tokens (`?data=<JWT>`, `/f/a/<token>`) encode the destination only on the sender's server, so the risky hop buys nothing for the opaque majority; (3) **cost/fragility** — network latency against the ~6-minute limit, quota burn, and a non-deterministic, hard-to-unit-test cleaning step.
- **Opaque-token resolution lives at the screening layer:** Claude resolves canonical job URLs by click-free **content-search**, never by following links. Offline, opaque trackers pass through unshrunk — accepted.
- **Accepted cost: no re-encoding of decoded separators.** A decoded destination's inner separators come back as bare `&` (from `%26`); we do not re-encode to `&amp;`. Harmless for the screening consumer (it reads text, not a browser).
- **Accepted cost: bare-text URL followed by a content entity** (`&nbsp;`, `&hellip;`, …) can be mis-harvested — `&` is ambiguous in an HTML body. Does not affect the href-based corpus; documented in [KNOWN_ISSUES §5](KNOWN_ISSUES.md), not fixed because excluding `&` from the harvest would truncate real raw-`&` trackers.
- **Implementation notes (review hardening).** The in-place swap is one position-based `String.replace` pass over the original body (not repeated `split`/`join`), so a freshly-inserted destination is never re-scanned and a URL that is a substring of another can't be corrupted by a later swap. Trailing punctuation is trimmed with a linear character walk, not an anchored `/[…]+$/` regex, which would backtrack O(n²) on a long punct run in a sender-controlled URL token.
- **Superseded: a dedicated extracted-links field.** Harvesting hrefs into a separate junk-filtered, deduped Airtable field was an earlier plan for the same tracker-bloat problem; the in-place cleanup above delivers real links to the pipeline with no separate field and no schema change. **Revisit when:** the screening pipeline wants a harvested, junk-filtered links list of its own.

The whole stage is pure and unit-testable (`tests/link-cleanup.test.js`, an end-to-end real-email fixture, and a mutation-checked wiring test). Parity preserved: emails with no trackers/utm produce exactly the regex-only `CleanText`.

## 4. Collector — HTML cleaning & no-library policy

The collector carries hand-written string logic: `CLEAN_REGEX` (HTML noise stripping), the offline link cleanup (`harvestUrls_` / `decodeEmbeddedDestination_` / `stripUtm_` / `cleanLinksInHtml_`), the single-child table-wrapper unwrap (`collapseTableWrappers_`), and the per-sender footer cutoff (`truncateAtFooter_`) — both below. Question raised during PR #6: wouldn't an established HTML/URL library be safer and less code?

**Decision: while the collector runs on Google Apps Script, keep the custom pure-function approach — no third-party HTML or URL library.**

1. **No dependency story on GAS without a build step.** `clasp` pushes raw files, and `.claspignore` ships exactly one (`gmail-collector.gs`, plus `appsscript.json`); there is no npm at deploy time. A library means bundling (webpack/esbuild → one large `.gs`) or vendoring — adding a bundler and a compile-before-push pipeline to a project that deliberately has none. It would also break the test harness's central trick: `tests/helpers/load-collector.js` runs the raw `.gs` in a `vm` *because* the file stays free of `require`/`module.exports`.
2. **Byte-stable cleaning output is the contract — pinned by the golden corpus, not a live Make comparison.** `CLEAN_REGEX` was ported **1:1 from the (now-retired) Make.com "Text parser" regex**, and its output is now locked by the golden corpus; a real DOM parser produces *different* output and would blow up those goldens. `URL`/`URLSearchParams` **normalise** (re-percent-encode, drop default ports, …), breaking the "byte-identical output when nothing changes" guarantee, and assume `&` separators where our hrefs are HTML-entity-encoded (`&amp;`) with original separators and untouched params deliberately preserved. Surgical string ops honour that byte-stability more simply than wrapping a library to defeat its normalisation.
3. **No viable built-in either.** GAS's `XmlService` chokes on real email HTML (almost never well-formed XML — that malformedness is *why* regex stripping exists). And GAS's V8 runtime most likely does **not** expose the WHATWG `URL`/`URLSearchParams` globals (web-platform APIs, not ECMAScript) — **unverified**; a two-line check in the GAS editor would confirm before any future URL-library consideration.
4. **The custom logic is compact and well-tested.** A few hundred lines of pure string functions across the cleaning stages (regex strip, link cleanup, table-wrapper unwrap, footer cutoff), covered by a `node --test` suite including a real-email corpus and mutation-checked guards — a modest, bounded maintenance surface. (Magnitude, not a pinned count — it grows a little with each cleaning slice; the inflection criteria below are what matter.)

**Rejected:** cheerio / parse5 (DOM parsing — breaks parity, needs bundling); `URL` / `URLSearchParams` (normalising — breaks parity; availability on GAS unverified).

**Accepted cost:** we own the parsing edge cases. This bit during PR #6 — an HTML injection via a decoded `</body>`, a trailing-punctuation ReDoS, a bare-text-URL entity absorption — all now pinned by regression tests, but the class of risk is ours to carry.

**Revisit when:** adopt the platform `URL` + a real HTML→text parser when either (a) the cleaning ambition grows materially — the backlog's **second cleaning pass** (tag→text, entity decode, block-boundary newlines) is genuinely hard to hand-roll well — or (b) the cleaning moves **off Apps Script** to a future Node/Python service, where the constraints above vanish. The inflection point is "the runtime changed" or "the ambition grew" — not "the code got fiddly". Those two tend to arrive together, and that is the natural moment to bring in libraries rather than retrofitting a bundler onto the single-file GAS collector now.

### Single-child table-wrapper unwrap (issue #13)

Job-alert senders wrap content in layout-only `<table><tr><td>` chains holding exactly one inner element, often several levels deep; after `CLEAN_REGEX` these skeletons carry zero information but still cost screening tokens on every email. **Decision: collapse them in `processMessage_` immediately after `CLEAN_REGEX`** — `collapseTableWrappers_` replaces a `<table>` whose content is exactly one `<tr>` (optionally via a single `<tbody>`) holding exactly one `<td>` containing exactly one element and no non-whitespace text with that element, repeated to fixpoint under a pass cap. Semantics ported from the retired v3 Python design §3.5 (BeautifulSoup `collapse_table_wrappers`) to a pure string/stack function, consistent with this section's no-library policy. Only `CleanText`/`CleanLength` reflect it; `HtmlLength` stays the original body length (Make-port parity).

- **Conservative by construction.** A quote-aware tokenizer + strict stack pairing: a table collapses only when its whole skeleton chain tokenizes into strictly matched open/close pairs — any unmatched tag, stray close, or stray text leaves it untouched, so malformed HTML degrades to a no-op, never a mangle. Content tables (multi-row, multi-cell, `th`, a `td` mixing text with elements) never match; the kept element is preserved verbatim (attributes included — `CLEAN_REGEX` strips a fixed attribute list, so `colspan`/`lang`/`title` etc. survive and tags are tokenized as `<td[^>]*>`, never assumed bare). With nothing to unwrap the output is **byte-identical** (the same parity contract as the link cleanup).
- **Measured win** (fixture corpus through the full pipeline, 2026-06-11): 17–40 wrappers and ~4.7–25% of `CleanText` removed per table-based email; ziprecruiter (div-based layout) is the pinned 0-wrapper/byte-identical case. Exact per-fixture values are golden-pinned in `tests/table-unwrap.test.js`, so a silent drop in the win is a visible regression.
- **Relationship to the backlog's "Second cleaning pass" (tag→text):** tag→text would subsume this saving wholesale; the unwrap is the safe incremental step available now. If/when tag→text lands, this pass either retires or becomes its pre-stage (`TODO.md` → Collector).
- Log lines (per-email + per-run rollup, real and DRY_RUN alike): [OPERATIONS — Collector: table-wrapper unwrap](OPERATIONS.md#collector-table-wrapper-unwrap).

### Per-sender footer cutoff (issue #14)

Sender footers carry the **riskiest links** in the corpus — one-click action endpoints (unsubscribe, "pause forever", 1-click feedback ratings) plus tracking-pixel CSS and legal boilerplate. §3 already rejected *following* links because of exactly these side-effect endpoints; cutting the footer removes them from the data the screening layer reads at all. Token saving is secondary but real (~0.5–11 KB of `CleanText` per mapped email in the fixture corpus). **Decision: an opt-in, per-sender footer cutoff (`truncateAtFooter_`) in `processMessage_`, immediately after the table-wrapper unwrap** — so the marker is matched against the fully-cleaned text (links → `CLEAN_REGEX` → unwrap → cutoff). As with the earlier stages, only `CleanText`/`CleanLength` reflect the cut; `HtmlLength` stays the original body length (Make-port parity), and an unmapped sender is a **byte-identical no-op**. Ported and upgraded from the retired v3 Python design §3.6 (`FOOTER_MARKERS`/`truncate_at_footer`), consistent with this section's no-library policy (a `lastIndexOf` + `slice`, no DOM).

- **Opt-in, keyed by registered domain.** `FOOTER_MARKERS` maps a registered domain → a marker (a literal string, or a `{text|urlPattern, mode}` object — see the marker modes below), **or an array of markers** (one per template — see *Multiple live templates* below), that begins that sender's footer; an unmapped sender is untouched and never alarms (GAS-side — the screening-side footer-freshness scan below is the same-day manual catch for that gap). Markers outlive sender addresses (whatjobs moved `mail.whatjobs.co.uk` → `mail.uk.whatjobs.com` in 2026 but kept its footer text), so the map is keyed by **registered domain**, matched by exact equality **or a dot-boundary suffix** (`domain === key || domain.endsWith('.' + key)`) — never a bare suffix (so `notwhatjobs.com` does not match a `whatjobs.com` key, and a `jobs.co.uk` sender is not swept up by some unrelated key). First matching key wins.
- **Marker modes — `text` (default) / `link` / `urlcut` (footer-cut-token-lead, 2026-06-20).** A string value is `text` mode (current behaviour, unchanged). The other two handle **token-lead footers**, where the per-recipient tracking token sits *before* the marker text, so a plain `text` cut (`slice(0, lastIndexOf(text))`) would cut the text but **leave the token** in the `<a href="…token…">` opening tag (the trap that deferred haystack/talentsource/applygateway). `{text, mode:'link'}` does `lastIndexOf(text)` then **snaps the cut back to the nearest preceding `<a>` open tag** (searched within the floor→marker window) so the leading anchor's token goes too; if there is no safe anchor in that window (the template lost/moved the `<a>`) it is a **`miss` (fail loud)**, never a text-index cut — a text cut would report `hit`, strip the visible footer text, and silently leave the per-recipient token in `CleanText` while raising neither the GAS miss alarm nor the screening tail scan (Codex F1, PR #42). `{urlPattern, mode:'urlcut'}` is for a footer whose only element is a per-recipient link with **no anchor text** (text mode has nothing to match — the cleaned body ends mid-`<a href>`): it cuts at the **last** `<a …href="…">` whose href matches `urlPattern`, at/after the floor. The floor check applies to the **resolved** cut index in every mode. Still no DOM — a `lastIndexOf`/regex anchor-scan + `slice`. Chosen over a **global tracker-token strip** (a second cleaning pass that nukes every `?token=`/JWT/UUID corpus-wide): that is the rejected Option B — broad, hard to keep from eating legitimate URLs, and unnecessary when the footer cut already removes the riskiest links; the per-sender map stays the surgical tool (the tag→text "second cleaning pass" backlog item is a *different*, orthogonal idea).
- **Match rule — `lastIndexOf` + a position floor.** Footers are terminal, so the cut is at the **last** occurrence of the marker (the same phrase leaking into a job description above is ignored), and only if that index sits at ≥ `FOOTER_POSITION_FLOOR` (0.5) of the text length; an earlier or absent match is a **miss**, not a cut. v3 semantics: the marker goes **with** the discarded tail (`slice(0, idx)`). Corpus footers start ≥ ~68% in (whatjobs, the earliest), so 0.5 holds with headroom (pinned by a test).
- **Multiple live templates per sender — a marker *array* (footer-multi-marker, 2026-06-21).** A `FOOTER_MARKERS` value may be an **array** of markers when one sender ships more than one footer template. `truncateAtFooter_` normalizes a scalar to a 1-element array (so the scalar path is **byte-identical** — back-compat) and `footerCutIndexMulti_` resolves the **earliest valid cut** across the array — each marker still floor-checked individually (the floor is per marker, never floated to whole-array scope). Footers are terminal, so when two markers resolve the **earlier** index is the true footer start: cut more, not less. **"First matching *key* wins" is unchanged** — this is multiple markers *under one key*, not multiple keys. First user: **NIJobs**, which has a **residual-element** case (not a `miss`). Its `"N new \"X\" jobs in United Kingdom"` **digest** footer carries two elements in order — `Change criteria for jobs by email` (~0.81, a per-recipient `click.nijobs.com` tracker) **then** `Manage all your subscriptions` (~0.83) — so the single `Manage all your subscriptions` marker already **hits** (it is the production cut point: `stored CleanLength == footerCutIndex_(A)` on three sampled rows) but **leaves the earlier `Change criteria` tracker as a ~511 B residual** in `CleanText` because the mapped marker isn't the *earliest* footer element. The fix is to **append an earlier marker** (`Change criteria for jobs by email`) to the domain's array; earliest-cut-wins removes the residual too (5422 → 5933 B). NIJobs' other template (recommendation / "picked for you") has **no** `Change criteria` line, so the array cuts it **identically** to the single marker (back-compat, pinned). This is **PII-hygiene, not an alarm fix** — a residual after a `hit` raises no GAS miss alarm; it was found by **inspecting already-cut `CleanText`**. The screening-side footer-freshness scan surfaces this as a **matched-key** footer signal and proposes **appending** a marker to the domain's array — **never** replacing, because the scheduled run sees only the post-cut `CleanText` and can't tell whether the existing marker is dead (drift) or still serving another template (residual); append is safe under both (a stale appended marker is a harmless −1, deleting a live one breaks that template). See the screening-scan bullet below / instructions §3/§8.
- **Markers are validated against stored `CleanText` byte-form, confirm-twice-or-drop.** Entities survive `CLEAN_REGEX`, so markers are chosen entity-free from the stored corpus, not from rendered email text; a domain that cannot be confirmed against ≥2 stored samples is dropped this slice. Stays deferred until an RFC822 raw-message export path (byte-preserving) exists; `jfw@` is also effectively n=1. Tracked in `TODO.md`.

- **Fixture-redaction style.** Per-recipient tokens (JWTs, UUIDs, SparkPost send-ids, base64url tracking tokens, numeric viewer ids) are replaced by a **length-neutral, readable** `TOKEN_REDACTED…A` filler; redaction is applied uniformly by token-shape (mixed-case+digit / JWT / UUID / long-hex), which over-redacts safely. The **recipient name** in greetings/`<title>` (e.g. `Hi <name>,`) is redacted length-neutrally too (Codex F2, PR #42 — it was the one PII class the opaque-token scan didn't cover). Verified by a no-leak scan: no recipient-email form, recipient name, JWT, UUID, or token-shaped run survives, and no `U+FFFD` survives **in the cleaned body** (the 1–2 `U+FFFD` in a QP sender's `<head>` are intentionally left as-is — hand-scrubbing them would make the capture less faithful — and are stripped by `CLEAN_REGEX` before the cleaned output regardless).
- **Fixture-capture fidelity (re-spec'd 2026-06-19; supersedes an exact-length check).** A new `email-*.html` fixture is validated as a *faithful* capture by **byte-identity**, not by matching the stored `CleanLength`. Two realities make an exact-length match impossible: (1) the stored Airtable `CleanLength` is **CRLF-era** (it includes in-text `\r`) while the fixture corpus is **LF-only** (the tests assert `!/\r/`), and (2) the Gmail capture path (`get_thread`, the only format the MCP exposes) returns the **QP-decoded** body, so a quoted-printable sender's `<head>` (stripped by `CLEAN_REGEX`) and per-recipient tokens (redacted) pick up `U+FFFD`/`\x17` decode artifacts that never reach the cleaned output. Faithfulness gate: `collapseTableWrappers_(clean(cleanLinksInHtml_(raw).html)).html` of the LF capture is **byte-identical** to the stored `CleanText` after CR-stripping the stored value **and** masking the same per-recipient token regions on both sides. **footer-cut-token-lead (2026-06-20) added two normalizations to this gate, both transport-only:** (a) **whitespace-collapse** — the Gmail MCP normalizes trailing inter-text whitespace differently from the live collector's Gmail-API path (e.g. `Hi Ivan, ` vs `Hi Ivan,`), so the compare collapses runs of `[ \t]` and trims around newlines; and (b) the gate accepts a faithful **prefix**, not just equality — for an already-mapped sender the *live* collector has already cut its footer (at the old marker, for the drift senders), so the stored `CleanText` is the pipeline output **truncated at that point**; the capture is faithful when stored is a (masked, ws-collapsed) prefix of the full pipeline output (equal for the previously-unmapped senders, strict prefix for the drift senders). The gate runs on the *unredacted* capture (proving the bytes are real); redaction is a separate length-neutral transform verified by a no-leak scan. Goldens (clean-regex byte-counts, unwrap counts, footer `bytesCut`) are **re-measured from the shipped redacted LF fixture** — never derived from the CRLF-era stored length. The QP `<head>` artifacts are left as-is (hand-scrubbing the head would make the capture *less* faithful / more "processed"); they are stripped before the cleaned output regardless.

**Template-change alarm (owner decision 2026-06-10: GAS failure email).** A mapped sender whose marker is absent or fails the position floor is a likely **template change**. The miss does **not** cut (no-op), logs a per-email warning, and is aggregated into an **end-of-run throw** — the same pattern as `upsertFailures`: every cut row is already committed, the throw only flips the execution to **Failed** so the GAS failure email fires and tells Ivan to update the marker. Ordering: when a sub-batch upsert also failed, the thrown error names the upsert failure first (the data-integrity event) **and folds in the footer-miss summary** — a miss in a sub-batch that *did* commit is already `collected` and would **not** recur, so the alarm must ride the same failure email rather than be suppressed (F1, PR #17). **DRY_RUN throws nothing** (logs the would-be misses). Unmapped senders never alarm **GAS-side** — the screening-side footer-freshness scan (below) is the same-day manual catch for that gap.

- **Accepted cost (owner-chosen):** with the ~30-min collector trigger, a changed template fails **~48 runs/day** until the marker is updated — accepted as the price of a loud, immediate signal over a silent parser break (v3 §9's "sender changes template, parser breaks silently" risk).
- **Rejected:** an Airtable `FooterMarkerMiss` flag written per-miss + a daily-report canary instead of throwing — quieter, but it defers the signal a full day and adds a schema field and a screening-side reader for a rare event. **Revisit when:** the failure-email alarm proves too noisy from legitimate template churn (then switch to the flag-and-report path).
- **Out of scope, recorded as a future option:** a volume/size-drop canary (alert when a sender's daily volume or mean `CleanText` size drops vs a 7-day average — v3 §9), which would catch a footer-grew-instead-of-vanished failure the marker miss can't.
- Log lines (per-email for mapped senders + per-run rollup, real and DRY_RUN alike): [OPERATIONS — Collector: per-sender footer cutoff](OPERATIONS.md#collector-per-sender-footer-cutoff), with the **marker-miss runbook**.

**Screening-side footer-freshness scan (complements this alarm; owner idea 2026-06-14, shipped post-M6.2 in `instructions/Claude_project_instructions.md` v2.2).** The GAS alarm above fires only for **mapped** senders whose marker drifts (a `miss`), and only as a run-failure email; **unmapped** senders are a byte-identical no-op the collector never flags. The daily screening run (Cowork-side, reading `CleanText` from RawEmails) closes that gap with a **two-case** classification: a footer signal still present at the tail of a row's `CleanText` is matched against `FOOTER_MARKERS` (no key matches the sender host → **new** → propose a fresh **registered-domain scalar** key; a key matches → **append** a marker to that key's value, converting a scalar to an array — **never replace**). The candidate (`'<domain>': '<phrase>'` / array form) is entity-free byte-form, terminal, past the 0.5 floor — surfaced same-day, no digging into Airtable/Gmail. **Why append-always for a matched key (Codex F1, PR #45):** the scheduled run has only the post-cut `CleanText` and **not** the collector's per-message `hit`/`miss`, so it **cannot** tell a *drift* (marker dead, footer uncut) from a *residual* (marker cut at a later point, earlier element left) — a `hit` can still end `CleanText` with a pre-marker unsubscribe endpoint, and a `miss` can leave only a short fragment; "how much remains" and "spot-check another row" (a cut *removes* the marker from `CleanText`) both fail. So it must not guess: append is safe under both branches (a stale appended marker is a harmless −1; replacing a still-live marker breaks the other template). A *total* drift (no array marker matches) is still caught loudly by the collector's `miss` → GAS email; pruning a confirmed-dead marker is **manual** housekeeping (verify deadness from a fresh Gmail/raw capture, not from `CleanText`). Shipped in instructions §3/§8 alongside the collector's array support. It is the **human-in-the-loop feeder** that keeps `FOOTER_MARKERS` current; fuzzy tail-spotting belongs with the screening agent, **not** GAS. It does **not** change collector behaviour — the alarm, the cutoff, and the marker map are untouched; adding a flagged marker stays the manual follow-on. Detection + output contract: `instructions/Claude_project_instructions.md` §3 (footer-freshness check) / §8 (alert). Operator action: [OPERATIONS — Screening: footer-freshness alert](OPERATIONS.md#screening-footer-freshness-alert).

## 5. Data model (Airtable)

- **Vacancies stays decisions-only** (applied/skipped/flagged) — no full vacancy inventory. **Rejected:** storing every parsed vacancy — the Airtable free-tier record cap; ~100 parsed vacancies/day would blow it in weeks.
- **Capacity budget — the free-plan record cap is per BASE, shared across all tables.** 1,000 records for RawEmails + Vacancies + everything else, crossed 2026-06-10 when a 3-week backlog sweep took the base to 1,017 (RawEmails 940 + Vacancies 77). Enforcement observation from that incident: Airtable's over-limit notification arrived 11:13, record creation was still succeeding ≥5 hours later — treat enforcement timing as **unspecified**, never as headroom (see [KNOWN_ISSUES §6](KNOWN_ISSUES.md)). Allocation: RawEmails ≤ 700 (purge high-water), leaving ≥ 300 for Vacancies growth and slack. A second free-plan constraint: the **API-call budget** of 1,000 calls/workspace/month (verified against Airtable's docs 2026-06-10) — batch everything; one-call-per-record patterns are unaffordable.
- **RawEmails is a transient queue, kept inside its share by the purge job** (`purgeRawEmails`, same script file, own nightly trigger — time: §7): when the record count exceeds `PURGE_HIGH_WATER` (700, runtime-tunable), delete the **oldest** eligible rows down to `PURGE_LOW_WATER` (500, tunable); eligible = `Status='Processed'` AND `CollectedAt` older than `PURGE_MIN_AGE_DAYS` (2), enforced **server-side** by the `filterByFormula` so `Status='New'` rows structurally cannot be deleted — emergency purges of unprocessed rows stay a manual/owner action. Capacity-triggered with an age floor, not age-triggered (supersedes the earlier "purge after ~7 days" sketch): the trigger is the budget, the age floor is the safety margin. Starvation (over high-water, 0 eligible — the *normal* state pre-M6.2) logs a warning; at `PURGE_EMERGENCY` (950) with 0 eligible the run throws so the failure email fires before writes get blocked. **Rejected:** CI-cron janitor — violates §7's "no CI-triggered execution, ever"; Airtable Automations — no native delete-record action, "Run a script" needs Team plan, and a UI-pasted script is unversioned. **Revisit when:** the M6.2 cutover starts producing `Processed` rows at volume, or the purge hits starvation near the cap.
- **Schema-as-code, additive-only.** `airtable/schema.json` (covering the two managed tables — RawEmails **and** the live **Vacancies** decisions store; the M5 `Vacancies_test` sandbox mirror was retired so `schema.json` and the live base stay in sync) + an idempotent apply script in CI (Meta API). The API cannot delete tables/fields or change field types — removals stay manual. Matching is **by field ID when present** (name is the fallback for any entry without an id), so a field renamed in the Airtable UI surfaces as a **rename-drift warning** (`schema.json says <name>, live is <liveName>`) instead of being re-created as a duplicate. `airtable/import-schema.js` (Meta API → `schema.json` merge, additive/idempotent, managed-table allowlist) backfills the live ids and produces diffable drift snapshots — run it before editing the schema. Both managed tables now carry live ids (RawEmails backfilled 2026-06-17), so the residual footgun is **future-only** — it returns for a newly added entry committed name-only until import-schema backfills its id: see [KNOWN_ISSUES §3](KNOWN_ISSUES.md).
- **Off-platform Vacancies backup (daily CSV → Google Drive).** Airtable has **no API-schedulable, off-site backup**: native snapshots are usage-based, UI-only, plan-bound and **in-platform** (they die with the base/account; a Team→Free billing lapse can purge them — confirmed against Airtable's docs 2026-06-20). **Vacancies is the only irreplaceable asset** (Ivan's Applied/Skipped decision history — dates, links, notes); RawEmails is regenerable from Gmail, so it is **not** backed up. A separate Apps Script file (`apps-script/vacancies-backup.gs`, `backupVacancies()` on its own daily trigger — §7) reads the whole Vacancies table (offset-paginated, rename-safe via `returnFieldsByFieldId=true`) and writes a dated `Vacancies_YYYY-MM-DD.csv` (RFC 4180, leading `recordId`/`createdTime`, all fields) into a fixed Drive folder, **replacing** same-day re-runs (idempotent, no duplicate). Fail-loud: an Airtable read failure after a light transient retry, or a **0-record result** (the empty-result guard — never clobber a good prior CSV with an empty one), **throws** rather than writing a partial/empty file. **Additive & self-contained:** new `.gs` + one Drive scope + docs; the collector's intake/screening/dedup/triggers are untouched, so a `git revert` of the file + scope + `.claspignore` line + docs fully removes it. The column order is mirrored from `schema.json` into the file (the runtime can't read `airtable/schema.json` — clasp pushes only `apps-script/**`) and a unit test holds the mirror in lockstep with the SSOT. **Rejected:** native Airtable snapshots (in-platform, plan-bound, not off-site — the very single-point-of-failure this guards against) and third-party backup SaaS (On2Air/ProBackup — recurring cost + another credential/vendor for a single-table need a 20-line script covers). **Restore caveat — DR ≠ "just restore", and the two recovery sources differ:** an Airtable **native snapshot** preserves table/field IDs, so only the **`baseId`** is repointed; a **CSV re-import** (this backup) assigns **new base, table *and* field IDs** plus inferred types, so it is a full **identity rebuild** — recreate the typed schema (also recreating RawEmails, since the CSV is Vacancies-only), import the data, capture the new IDs, then repoint **every** stale id (base + Vacancies/RawEmails table ids + the field ids pinned in `instructions`, `BACKUP.VACANCIES_FIELDS`, and `schema.json`, where a stale id otherwise spawns a duplicate table on apply) before redeploying. Both restore paths, split step-by-step: [OPERATIONS](OPERATIONS.md#vacancies-backup-off-platform-dr). **Revisit when:** Airtable ships an API-schedulable export, or a second table becomes irreplaceable.

## 8. Slice dispatch — Architect → Implementer handoff (tooling)

> **PARKED as of 2026-06-14.** The *cloud-dispatch automation* below (`dispatch-slice.sh` + the launchd watcher) is archived under `scripts/slice-passing-parked/`, not in active use: a CLI-launched `claude --remote` can't push in this environment (HTTP 401). The kept flow is the human-run `/run-slice`. Findings, the blocker, and revival steps: [slice-passing-parked/README](../scripts/slice-passing-parked/README.md). The decisions are retained below as the design record.

- **The Architect (Claude Cowork) hands each slice to the Implementer (Claude Code) as a GUI-visible session.** The Architect stages a slice under the gitignored `issue-drafts/<slice>/` (the spec `slice-prompt-<slice>.md` plus any fixtures); `scripts/slice-passing-parked/dispatch-slice.sh <slice>` cuts a branch off `origin/main` (prefix = work type, declared in the spec), force-stages that folder onto it, pushes, and launches the session with a working-agreement preamble (`scripts/slice-passing-parked/slice-agent-preamble.md`) prepended to the spec. It never dispatches from `main` and never merges. Mechanics, limitations, and the prompt template: [SLICE_DISPATCH](../scripts/slice-passing-parked/SLICE_DISPATCH.md), [SLICE_PROMPT_TEMPLATE](SLICE_PROMPT_TEMPLATE.md).
- **Decided: `claude --remote` (cloud), not `claude -p` (local headless).** A `claude -p` session is invisible in the desktop GUI — no Code-tab card, nothing on claude.ai/code or iOS — so the owner can't watch the slice run; `claude --remote` cloud sessions are GUI-visible and stream progress. They also bill against the normal subscription rate limits with no separate compute charge, unlike `claude -p` (which draws the separate Agent SDK credit — see [SLICE_DISPATCH](../scripts/slice-passing-parked/SLICE_DISPATCH.md#billing)). **Rejected `claude -p` for this handoff;** revisit only if a non-interactive/CI use case (no human watching) appears, and then add it behind a flag rather than replacing `--remote`.
- **Decided: Pattern A fixture staging (force-commit the staging dir), not prompt-inlining.** A cloud session runs in a fresh VM that clones the repo from GitHub and can't read local files, and `issue-drafts/` is gitignored — so the dispatcher force-commits `issue-drafts/<slice>/` onto the feature branch before pushing, so the clone carries the spec *and* fixtures. Inlining fixture *contents* into the prompt was rejected: byte-exactness (LF, the U+FFFD head quirk, per-recipient token redaction) doesn't survive a prompt string — let git carry the bytes. The Implementer then moves fixtures into their committed home and removes the staging dir, so after a squash-merge `main` shows only the fixtures in their home, never `issue-drafts/`. Fixture-heavy slices can instead run a **local** Code-tab session that reads `issue-drafts/<slice>/` straight off disk (no force-stage).
- **The launchd watcher ships opt-in and disabled.** `scripts/slice-passing-parked/watch-slices.sh` + `scripts/slice-passing-parked/com.kathelix.slice-dispatch.plist` are a GO-sentinel poll loop for hands-off auto-dispatch; they are **not** `launchctl load`-ed by default. Enable per the `watch-slices.sh` header (the canonical launchctl runbook) only with eyes on it, and point it at a dedicated worktree so auto-branching can't collide with manual work.
