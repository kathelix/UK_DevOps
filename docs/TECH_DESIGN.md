# Technical Design

_Living document — edited in place, history lives in git. One section per design area; each records the current design, its rationale, **Rejected:** alternatives with why, and **Revisit when:** triggers, so settled questions aren't re-litigated._

## 1. Architecture overview

Gmail filters funnel all job-alert email under one label; a Google Apps Script collector picks it up on a time trigger (cadence: §7), cleans it offline, and stores clean text in Airtable (RawEmails queue + Vacancies decisions); a scheduled Claude (Cowork) run screens the queue against versioned instructions, writes decisions, and reports to Ivan. The flow diagram and component table live in the [README](../README.md) — not duplicated here.

## 2. Collector — fetch & write pipeline

- **Sub-batch pipeline with a forward-progress guarantee.** The run is an interleaved fetch → upsert → label loop over sub-batches (`CONFIG.SUB_BATCH_SIZE = 5`), each committed before the next — not a two-phase fetch-all-then-write-all. The runtime budget (`MAX_RUNTIME_MS`, against the ~6-minute Apps Script execution limit) is checked once per sub-batch, and the first sub-batch always runs: every run commits at least one sub-batch, and a timeout or crash loses at most one. **Rejected:** separate budget breaks in a two-phase fetch/write structure — under sustained slow-fetch latency the write-phase guard tripped before writing anything, re-collecting the same head-of-queue messages with no forward progress. Guarded by the forward-progress test in `tests/collect-loop.test.js`.
- **Write-then-label ordering.** The Airtable write happens before the Gmail label; a crash in between re-presents the message next run. No data loss by design — the dedupe below makes the retry safe.
- **Dedupe on retry.** Airtable upsert (`performUpsert` on `MessageId`) rather than pre-checking existing rows. **Revisit when:** rows become per-vacancy instead of per-email — switch the dedupe key to `gmailMessageId + urlHash`.
- **Read-side: transient-retry then poison isolation — symmetric with the write side.** The per-message Gmail read mirrors the write side's transient-vs-deterministic split, in **two narrow `try/catch` stages** with separate failure handling. (1) `Gmail.Users.Messages.get` runs through `gmailReadWithRetry_`, which retries a **transient** read blip (`[1s, 2s, 4s]`, budget-aware) and, if it persists, leaves the message **uncollected** (unlabelled → re-presents next run, idempotent on the `MessageId` upsert) and **never** `make-failed` — a transient read is not poison. A persistent read fails **loud** (run ends Failed) so a Gmail-read outage can't silently stall the queue. (2) A deterministic **parse** failure (`headerMap_`/`processMessage_` — a malformed MIME tree) still `make-failed`s the message (quarantine + MIME-tree forensics log), exactly as before, and does **not** trip the read canary (a quarantined poison is *handled*, not an un-collected read). Splitting the formerly-**single** `try/catch` into two is the fix for a **latent data-loss bug**: the old catch labelled **any** throw — including a transient `get` `429`/`5xx`/network blip — `make-failed`, evicting a perfectly good message from `CONFIG.QUERY` *permanently*. The once-per-run `Gmail.Users.Messages.list` is wrapped the same way (a transient blip heals; a persistent failure propagates → run Failed), removing the spurious single-blip run failure. A run hitting any combination of **read failure**, **upsert failure**, and **footer-marker miss** throws **one** error naming *every* present signal (precedence: reads, then writes, then the footer alarm), so no co-occurring signal is swallowed (F1, PR #17). The read-failure alarm also fires in **`DRY_RUN`** — DRY_RUN does real reads, so a persistent read outage must fail loud there too or dry-run validation silently misses it (Codex F1[P2], PR #25); the write-path signals do **not** (an upsert never runs in DRY_RUN, and a footer miss is a side-effect-only template alarm deliberately suppressed in dry runs). **Out of scope (Option B):** parsing the HTTP status out of the thrown error to skip a genuine `404` — it depends on the Gmail advanced service's runtime error shape (`GoogleJsonResponseException`), which the stubbed Node harness can't verify ("a stubbed runtime hides runtime-only behavior"), so the read retries on **any** throw; a `404`-skip is a possible future follow-up gated on a real-runtime probe, only if alert noise appears (owner decision 2026-06-15). **Rejected:** merging the two retry wrappers — different transport contracts (Gmail throws on non-2xx and returns no response; Airtable uses `muteHttpExceptions` + `getResponseCode()`); they share only the backoff constant and the budget predicate.
- **Write-side per-record isolation — any failed sub-batch (poison or transient).** A single record-specific failure fails the whole all-or-nothing sub-batch PATCH — a deterministic `4xx` reject (e.g. `422` validation) **or** a `5xx`/transport blip that trips on one record — so without isolation the good siblings are never `make-collected` and are re-fetched on *every* run (each ending Failed) until the sub-batch composition happens to change. `airtableUpsert_` returns the numeric HTTP code; `attemptUpsert_` classifies it `ok` (`200`) / `transient` (`429`/`5xx`/network throw, `isTransientWriteFailure_`) / `poison` (any other `4xx`). **Any non-`ok` batch** is re-sent record-by-record on its own PATCH (through `airtableUpsert_`, so the transient-retry wrapper below composes with the isolation), and each record takes its per-record outcome: **`200`** → `make-collected`, unblocking a healthy sibling instead of stranding it behind one bad record; **`transient`** → left uncollected, retried next run, **never** `make-failed` on a single failure (a transient is not poison; a *record-specific* one that persists is capped across runs — see the next bullet); **`poison`** (`4xx`) → `make-failed` **only if ≥1 sibling upserted `200`** that run (the **quarantine guard** — proof the endpoint/auth/schema is healthy and the reject is record-specific). Removing the old `transient` early-`continue` — which left a whole transient sub-batch uncollected on every run — is what unblocks the healthy siblings: a record-specific `5xx` no longer holds them hostage. **Systemic failure unchanged:** if **zero** records succeed — every record `4xx` (a systemic `401`/`404`/`422 UNKNOWN_FIELD` from bad auth, wrong endpoint, or schema drift) **or** an all-records transient outage (no healthy sibling) — it quarantines **nothing for a fresh (never-struck) record**, leaves the whole sub-batch uncollected, and fails loud, so a deploy mistake or an outage can never *mass*-quarantine the queue (`make-failed` excludes a message from `CONFIG.QUERY` permanently). The one bounded exception is the **struck-then-outage residual** — a *transient* record that already earned record-specific strikes can finish capping to `make-failed` even in such a run (sticky record-specificity; see the cap bullet below) — never a fresh record, so never a mass quarantine. `upsertFailures` counts **once per sub-batch** that leaves anything stuck (not per record — F-P3), and the run ends Failed so the GAS failure email fires. **Accepted cost:** during a true systemic outage every failed sub-batch now individually re-sends (≤ `SUB_BATCH_SIZE` records × the retry schedule), bounded by `MAX_RUNTIME_MS` (the budget predicate stops the run before it burns the execution limit) — the same cost profile the poison path already had; an outage isolation fast-path is a possible future optimisation, deliberately not added here. **Accepted limitation:** a poison record *alone* in its sub-batch has no sibling to prove record-specificity, so it stays uncollected and keeps alerting until it re-batches with others or a human handles it — conservative by design. **Foundation for the repeatedly-transient cap:** the per-record `transient` outcome this exposes — a record-specific write that is stuck but not poison — is where the **repeatedly-transient write cap** (next bullet) hangs its cross-run strike counter. **Rejected (still standing for the *deterministic* case):** a cross-run per-message failure counter that would `make-failed` a *deterministic* reject after N failed runs — persistent per-message state, it does **not** unblock the good siblings in the same PATCH, and it is slower to quarantine than the within-run individual isolation above, which is record-specific and immediate (owner decision 2026-06-12). That same counter **does** fit the **repeatedly-transient** case — a stuck transient record has no within-run remedy, since retrying it only fails again — so it is **implemented** there (next bullet), built on the isolation prerequisite this bullet landed.
- **Repeatedly-transient write cap — `make-failed` after N record-specific strikes.** A record whose *own* PATCH trips a `transient` outcome on every run is otherwise retried **forever** and fails the run **every** run (the unbounded `upsertFailures` alarm storm the backlog flagged): the per-record isolation above unblocks its healthy *siblings* but never caps the stuck record itself. So each **record-specific** transient strikes a **cross-run counter**, gated **`anyHealthy || priorStrikes > 0`**: the **first** strike needs a same-run healthy sibling (proof the system is up and the failure is this record's), but once a message carries a strike the counter is **sticky** — its subsequent **solo** failures keep striking with no fresh sibling required. Stickiness is load-bearing: #26 make-collects the healthy siblings out of `CONFIG.QUERY`, so a genuinely-stuck record is **alone** on every later run, and a per-run-only health proof would freeze its counter at the run-1 value so the cap is never reached (Codex F1, PR #27). At `MAX_TRANSIENT_WRITE_RETRIES` (`CONFIG` default **5** ≈ 2.5 h of sustained record-specific failure at the ~30-min cadence; runtime-tunable via `getIntProp_`, bounds `[1, 100]`, like `MAX_MESSAGES`) **consecutive** strikes the record is `make-failed` (reusing `FAILED_LABEL_NAME` — no new label) so it stops re-presenting, with one final quarantine notification. **The counter lives in Script Properties, one integer per message keyed `wretry:<messageId>`** — it cannot live in Airtable, because the failing write means the message has no row (owner decision 2026-06-15). It is **loaded once per run** into an in-memory map (so clear-on-success doesn't fire a `deleteProperty` for every collected message — only the rare struck-then-recovered one), mutated in memory, and persisted immediately (`setProperty` on a strike, `deleteProperty` on clear/quarantine). **Strike semantics are consecutive:** any successful upsert of that message clears its counter (the `ok` collect paths, whole-batch **and** per-record); a systemic-outage run is **neutral** (neither strikes nor clears). **The mass-quarantine guard holds for the bulk of the queue:** a **fresh** (never-struck) message in a systemic outage **never** strikes (no first-strike sibling proof), so a multi-hour Airtable outage cannot quarantine fresh traffic. **Two accepted residuals** (owner-ratified 2026-06-15, resolving Codex F1 — Option 1 "sticky record-specificity"): (1) **struck-then-outage** — a record that already earned ≥1 genuine record-specific strike and then faces a *sustained* systemic outage while solo can accrue to the cap and be quarantined; bounded to already-suspicious records (**never** fresh ones, so never a *mass* quarantine — at most the handful currently near N), and recoverable (remove the `make-failed` label to retry); the quarantine notice + the OPERATIONS triage make it observable. (2) **solo-from-start never caps** — a record alone in the queue for its entire failing lifetime never earns a first strike, so it is never capped (same as the prior unbounded behaviour for that ultra-rare case; requiring first-strike sibling-proof is the deliberate price of never quarantining a fresh message during an outage). A quarantined record is **handled** (like the poison quarantine), so it does **not** count toward `upsertFailures`; it raises its **own** fail-loud signal instead — the end-of-run throw names every present signal in precedence **reads → upserts → quarantines → footer** (F1, none swallowed). **`DRY_RUN` does no strike, no quarantine, no Script Properties write/delete** — the whole upsert/isolation block is short-circuited before any of it (the map may be *loaded*, a read, but is never mutated); stated and tested explicitly rather than inherited (Codex F1[P2], PR #25). The `setProperty`/`deleteProperty` calls sit **outside** any upsert `try`, so a Script Properties error surfaces with its own stack rather than being masked as a transient write failure (narrow-catch, F-P2). **`make-processing` is deliberately not set** (the original Make scenario never set it either): single-flight + the idempotent write-then-label already give crash-safety, so a mid-flight marker would add persistent state for no benefit (the stale `-label:…/make-processing` token in `CONFIG.QUERY` is a harmless no-op, left for a separate cleanup). **Accepted cost:** a message manually deleted from Gmail while struck orphans a few-byte `wretry:` key — rare, bounded by Script Properties' generous cap; a GC sweep is a possible future follow-up, not this slice. **Rejected:** a distinct quarantine label (`make-stuck`) — reuse `make-failed`; time-based (rather than run-count) strike semantics. **Revisit when:** the quarantine alarm proves too noisy, or orphaned `wretry:` keys accumulate enough to warrant a sweep.
- **Transient-retry/backoff wrappers — heal a blip within the run.** Two sibling wrappers share the backoff constant (`CONFIG.RETRY_BACKOFF_MS`; `[1s, 2s, 4s]` = 3 retries, 4 attempts) and the budget predicate, but stay **separate** for their different transport contracts: `gmailReadWithRetry_` for the idempotent Gmail **reads** (above — retries on *any* throw, no status parsing) and `airtableFetchWithRetry_` for the Airtable **writes** (this bullet). Every Airtable `UrlFetchApp.fetch` goes through `airtableFetchWithRetry_`, which retries a **transient** outcome — HTTP `429`, any `5xx` (`isTransientWriteFailure_`), or a thrown transport exception — with `[1s, 2s, 4s]` backoff (`CONFIG.RETRY_BACKOFF_MS`; 3 retries, 4 attempts). A `200` or a **deterministic 4xx** (`400`/`401`/`404`/`422`) passes straight through (retrying a validation/auth reject only burns budget). The wrapper sits **below** the per-record isolation contract above: callers see the *final* code after retries, so a transient that persists still classifies `transient` — its record left for next run, fail-loud, via the isolation above — and the quarantine guard is unchanged. The **idempotent `MessageId` upsert** makes a retried PATCH safe — no duplicate rows. **Budget-aware:** on the collector path the wrapper is threaded a predicate `(ms) => isOverRuntimeBudget_(startMs, now()+ms)` and **skips a backoff sleep that would cross `MAX_RUNTIME_MS`**, returning the last code rather than risk a hard ~6-minute kill mid-write — the forward-progress guarantee and the per-sub-batch budget check are preserved. The sleep fn, clock, and backoff schedule are injectable so unit tests verify the sequence without real sleeps (`tests/reliability-helpers.test.js`); recovery-composes-with-isolation is pinned in `tests/collect-loop.test.js`. **`DELETE` is the exception — `retryOnThrow:false`:** re-deleting an already-gone id returns `404 MODEL_ID_NOT_FOUND` (verified against the Airtable API), so a DELETE retried after a *transport throw* (where a prior success can't be ruled out) could 404 a delete that actually landed and fail the run spuriously. So deletes retry on **`429`/`5xx` only** (a server-side reject removed nothing — safe), and a transport throw propagates on the first attempt (pre-wrapper behaviour). The purge path gets **no budget predicate** — its per-run delete volume is small (nightly delta), so the worst-case total backoff stays short; and a purge is non-critical and self-correcting if it ever did hit the limit (committed deletes are durable; the next night re-counts). **Rejected:** retrying deletes on a transport throw (the `404`-on-phantom-success risk); making the backoff schedule a runtime Script Property (a `CONFIG` constant is enough — tune and redeploy). **Revisit when:** a large post-M6.2 catch-up purge routinely deletes hundreds of rows per run — then add a budget predicate to the purge path too (owner/architect decision 2026-06-12).
- **Single-flight.** A `LockService` guard (`tryLock(0)`, exit cleanly if held, release in `finally`) prevents overlapping scheduled runs — duplicate writes and label races.
- **Runtime controls, not redeploys.** `MAX_MESSAGES` (fetch cap; `0` = pause switch), `MAX_TRANSIENT_WRITE_RETRIES` (the repeatedly-transient quarantine cap, `[1,100]`; above), and `DRY_RUN` (log would-be writes/labels, touch nothing) are Script Properties read at runtime. Semantics and procedure: [OPERATIONS — Collector: routine procedures](OPERATIONS.md#collector-routine-procedures).

## 3. Collector — offline link cleanup

Job-alert emails wrap links in tracking redirectors, which bloats `CleanText` (measured: ~10 KB of a 19.4 KB body was tracking-URL base64) and hides real job URLs from screening.

**Decision: the collector cleans links entirely offline — it makes no network calls of any kind** (no `UrlFetchApp`, no fetching/following/probing). Before `CLEAN_REGEX`, for every URL in the body (both `href="…"` values and bare-text URLs) it (a) decodes an embedded destination and (b) strips `utm_*` analytics params, swapping the cleaned form in place. Operational description and the per-run `Links: decoded=N utm_stripped=M bytes_saved=B` log line (the only output — no Airtable schema field): [OPERATIONS — Collector: offline link cleanup](OPERATIONS.md#collector-offline-link-cleanup).

- **Value-guard only — no param-name allow-list, no tracker-host list.** The embedded destination is the **first** query param in document order whose URL-decoded value is itself an absolute `http(s)` URL or an absolute path (`/…`, prepending the tracker's scheme+host). **Rejected:** a curated list of redirect param names (`url`, `u`, `dest`, `redirect_uri`, …) — a maintenance burden that silently misses any name not enumerated (`redirect_url`, `dest_url`, `r2`, …); the "value must be a URL/path" guard is zero-maintenance and future-proof. **Accepted cost:** if a tracker carries a non-destination URL-valued param *before* the real one in document order (e.g. `?img=https://cdn/logo.png&url=…`), the wrong value is picked — rare in click-trackers, deliberate trade-off (decided with Ivan).
- **`HtmlLength` parity.** `HtmlLength` stays the **original** body length (parity with Make's `length(1.htmlBody)`); only `CleanText`/`CleanLength` reflect the cleanup. With neither an embedded destination nor a `utm_` param present, the transform is a **byte-identical no-op**.
- **Rejected: network redirect resolution** (`UrlFetchApp`, hop on `3xx` `Location`, cap per run — prototyped in PR #5, closed unmerged). Judged both risky and incomplete: (1) **side-effect risk** — tracker links are not all idempotent GETs; some are one-click unsubscribe or 1-click-apply endpoints, and even a `HEAD`/no-body hop can trip them; (2) **incompleteness** — opaque tracker tokens (`?data=<JWT>`, `/f/a/<token>`) encode the destination only on the sender's server, so the risky hop buys nothing for the opaque majority; (3) **cost/fragility** — network latency against the ~6-minute limit, quota burn, and a non-deterministic, hard-to-unit-test cleaning step.
- **Opaque-token resolution lives at the screening layer:** Claude resolves canonical job URLs by click-free **content-search**, never by following links. Offline, opaque trackers pass through unshrunk — accepted.
- **Accepted cost: no re-encoding of decoded separators.** A decoded destination's inner separators come back as bare `&` (from `%26`); we do not re-encode to `&amp;`. Harmless for the screening consumer (it reads text, not a browser).
- **Accepted cost: bare-text URL followed by a content entity** (`&nbsp;`, `&hellip;`, …) can be mis-harvested — `&` is ambiguous in an HTML body. Does not affect the href-based corpus; documented in [KNOWN_ISSUES §5](KNOWN_ISSUES.md), not fixed because excluding `&` from the harvest would truncate real raw-`&` trackers.
- **Implementation notes (review hardening).** The in-place swap is one position-based `String.replace` pass over the original body (not repeated `split`/`join`), so a freshly-inserted destination is never re-scanned and a URL that is a substring of another can't be corrupted by a later swap. Trailing punctuation is trimmed with a linear character walk, not an anchored `/[…]+$/` regex, which would backtrack O(n²) on a long punct run in a sender-controlled URL token.
- **Superseded: a dedicated extracted-links field.** Harvesting hrefs into a separate junk-filtered, deduped Airtable field was an earlier plan for the same tracker-bloat problem; the in-place cleanup above delivers real links to the pipeline with no separate field and no schema change. **Revisit when:** the screening pipeline wants a harvested, junk-filtered links list of its own.

The whole stage is pure and unit-testable (`tests/link-cleanup.test.js`, an end-to-end real-email fixture, and a mutation-checked wiring test). Parity preserved: emails with no trackers/utm produce exactly the regex-only `CleanText`.

## 4. Collector — HTML cleaning & no-library policy

The collector carries hand-written string logic: `CLEAN_REGEX` (HTML noise stripping), the offline link cleanup (`harvestUrls_` / `decodeEmbeddedDestination_` / `stripUtm_` / `cleanLinksInHtml_`), the single-child table-wrapper unwrap (`collapseTableWrappers_`), and the per-sender footer cutoff (`truncateAtFooter_`) — both below. Question raised during PR #6: wouldn't an established HTML/URL library be safer and less code?

**Decision: while the collector runs on Google Apps Script, keep the custom pure-function approach — no third-party HTML or URL library.**

1. **No dependency story on GAS without a build step.** `clasp` pushes raw files, and `.claspignore` ships exactly one (`gmail-collector.gs`, plus `appsscript.json`); there is no npm at deploy time. A library means bundling (webpack/esbuild → one large `.gs`) or vendoring — adding a bundler and a compile-before-push pipeline to a project that deliberately has none. It would also break the test harness's central trick: `tests/helpers/load-collector.js` runs the raw `.gs` in a `vm` *because* the file stays free of `require`/`module.exports`.
2. **Byte-parity with Make is the contract during the parallel-run cutover.** `CLEAN_REGEX` is a 1:1 port of the Make.com "Text parser" regex; a real DOM parser produces *different* output and would blow up the golden corpus. `URL`/`URLSearchParams` **normalise** (re-percent-encode, drop default ports, …), breaking the "byte-identical output when nothing changes" guarantee, and assume `&` separators where our hrefs are HTML-entity-encoded (`&amp;`) with original separators and untouched params deliberately preserved. Surgical string ops honour parity more simply than wrapping a library to defeat its normalisation.
3. **No viable built-in either.** GAS's `XmlService` chokes on real email HTML (almost never well-formed XML — that malformedness is *why* regex stripping exists). And GAS's V8 runtime most likely does **not** expose the WHATWG `URL`/`URLSearchParams` globals (web-platform APIs, not ECMAScript) — **unverified**; a two-line check in the GAS editor would confirm before any future URL-library consideration.
4. **The custom logic is compact and well-tested.** A few hundred lines of pure string functions across the cleaning stages (regex strip, link cleanup, table-wrapper unwrap, footer cutoff), covered by a `node --test` suite including a real-email corpus and mutation-checked guards — a modest, bounded maintenance surface. (Magnitude, not a pinned count — it grows a little with each cleaning slice; the inflection criteria below are what matter.)

**Rejected:** cheerio / parse5 (DOM parsing — breaks parity, needs bundling); `URL` / `URLSearchParams` (normalising — breaks parity; availability on GAS unverified).

**Accepted cost:** we own the parsing edge cases. This bit during PR #6 — an HTML injection via a decoded `</body>`, a trailing-punctuation ReDoS, a bare-text-URL entity absorption — all now pinned by regression tests, but the class of risk is ours to carry.

**Revisit when:** adopt the platform `URL` + a real HTML→text parser when either (a) the cleaning ambition grows materially — the backlog's **second cleaning pass** (tag→text, entity decode, block-boundary newlines) is genuinely hard to hand-roll well — or (b) the cleaning moves **off Apps Script** to a future Node/Python service, where the constraints above vanish. The inflection point is "the runtime changed" or "the ambition grew" — not "the code got fiddly". Those two tend to arrive together, and that is the natural moment to bring in libraries rather than retrofitting a bundler onto the single-file GAS collector now.

### Single-child table-wrapper unwrap (issue #13)

Job-alert senders wrap content in layout-only `<table><tr><td>` chains holding exactly one inner element, often several levels deep; after `CLEAN_REGEX` these skeletons carry zero information but still cost screening tokens on every email. **Decision: collapse them in `processMessage_` immediately after `CLEAN_REGEX`** — `collapseTableWrappers_` replaces a `<table>` whose content is exactly one `<tr>` (optionally via a single `<tbody>`) holding exactly one `<td>` containing exactly one element and no non-whitespace text with that element, repeated to fixpoint under a pass cap. Semantics ported from the retired v3 Python design §3.5 (BeautifulSoup `collapse_table_wrappers`) to a pure string/stack function, consistent with this section's no-library policy. Only `CleanText`/`CleanLength` reflect it; `HtmlLength` stays the original body length (Make parity).

- **Conservative by construction.** A quote-aware tokenizer + strict stack pairing: a table collapses only when its whole skeleton chain tokenizes into strictly matched open/close pairs — any unmatched tag, stray close, or stray text leaves it untouched, so malformed HTML degrades to a no-op, never a mangle. Content tables (multi-row, multi-cell, `th`, a `td` mixing text with elements) never match; the kept element is preserved verbatim (attributes included — `CLEAN_REGEX` strips a fixed attribute list, so `colspan`/`lang`/`title` etc. survive and tags are tokenized as `<td[^>]*>`, never assumed bare). With nothing to unwrap the output is **byte-identical** (the same parity contract as the link cleanup).
- **Measured win** (fixture corpus through the full pipeline, 2026-06-11): 17–40 wrappers and ~4.7–25% of `CleanText` removed per table-based email; ziprecruiter (div-based layout) is the pinned 0-wrapper/byte-identical case. Exact per-fixture values are golden-pinned in `tests/table-unwrap.test.js`, so a silent drop in the win is a visible regression.
- **Relationship to the backlog's "Second cleaning pass" (tag→text):** tag→text would subsume this saving wholesale; the unwrap is the safe incremental step available now. If/when tag→text lands, this pass either retires or becomes its pre-stage (`TODO.md` → Collector).
- Log lines (per-email + per-run rollup, real and DRY_RUN alike): [OPERATIONS — Collector: table-wrapper unwrap](OPERATIONS.md#collector-table-wrapper-unwrap).

### Per-sender footer cutoff (issue #14)

Sender footers carry the **riskiest links** in the corpus — one-click action endpoints (unsubscribe, "pause forever", 1-click feedback ratings) plus tracking-pixel CSS and legal boilerplate. §3 already rejected *following* links because of exactly these side-effect endpoints; cutting the footer removes them from the data the screening layer reads at all. Token saving is secondary but real (~0.5–11 KB of `CleanText` per mapped email in the fixture corpus). **Decision: an opt-in, per-sender footer cutoff (`truncateAtFooter_`) in `processMessage_`, immediately after the table-wrapper unwrap** — so the marker is matched against the fully-cleaned text (links → `CLEAN_REGEX` → unwrap → cutoff). As with the earlier stages, only `CleanText`/`CleanLength` reflect the cut; `HtmlLength` stays the original body length (Make parity), and an unmapped sender is a **byte-identical no-op**. Ported and upgraded from the retired v3 Python design §3.6 (`FOOTER_MARKERS`/`truncate_at_footer`), consistent with this section's no-library policy (a `lastIndexOf` + `slice`, no DOM).

- **Opt-in, keyed by registered domain.** `FOOTER_MARKERS` maps a registered domain → the literal marker string that begins that sender's footer; an unmapped sender is untouched and never alarms. Markers outlive sender addresses (whatjobs moved `mail.whatjobs.co.uk` → `mail.uk.whatjobs.com` in 2026 but kept its footer text), so the map is keyed by **registered domain**, matched by exact equality **or a dot-boundary suffix** (`domain === key || domain.endsWith('.' + key)`) — never a bare suffix (so `notwhatjobs.com` does not match a `whatjobs.com` key, and a `jobs.co.uk` sender is not swept up by some unrelated key). First matching key wins.
- **Match rule — `lastIndexOf` + a position floor.** Footers are terminal, so the cut is at the **last** occurrence of the marker (the same phrase leaking into a job description above is ignored), and only if that index sits at ≥ `FOOTER_POSITION_FLOOR` (0.5) of the text length; an earlier or absent match is a **miss**, not a cut. v3 semantics: the marker goes **with** the discarded tail (`slice(0, idx)`). Corpus footers start ≥ ~68% in (whatjobs, the earliest), so 0.5 holds with headroom (pinned by a test).
- **Markers are validated against stored `CleanText` byte-form, confirm-twice-or-drop.** Entities survive `CLEAN_REGEX`, so markers are chosen entity-free from the stored corpus, not from rendered email text; a domain that cannot be confirmed against ≥2 stored samples is dropped this slice. Confirm-before-pin caught a bad seed: reed's proposed `Manage your job alerts` (v3) is **absent** from the stored reed template — the real footer reads "…manage your contact preferences or unsubscribe", so the marker was corrected (it cuts 1,311 B, matching the issue's own research evidence). `nexxt.com` was dropped because the capture transport corrupts its quoted-printable body (revisit with the unquoted-attribute cleanup issue).

**Template-change alarm (owner decision 2026-06-10: GAS failure email).** A mapped sender whose marker is absent or fails the position floor is a likely **template change**. The miss does **not** cut (no-op), logs a per-email warning, and is aggregated into an **end-of-run throw** — the same pattern as `upsertFailures`: every cut row is already committed, the throw only flips the execution to **Failed** so the GAS failure email fires and tells Ivan to update the marker. Ordering: when a sub-batch upsert also failed, the thrown error names the upsert failure first (the data-integrity event) **and folds in the footer-miss summary** — a miss in a sub-batch that *did* commit is already `make-collected` and would **not** recur, so the alarm must ride the same failure email rather than be suppressed (F1, PR #17). **DRY_RUN throws nothing** (logs the would-be misses). Unmapped senders never alarm.

- **Accepted cost (owner-chosen):** with the ~30-min collector trigger, a changed template fails **~48 runs/day** until the marker is updated — accepted as the price of a loud, immediate signal over a silent parser break (v3 §9's "sender changes template, parser breaks silently" risk).
- **Rejected:** an Airtable `FooterMarkerMiss` flag written per-miss + a daily-report canary instead of throwing — quieter, but it defers the signal a full day and adds a schema field and a screening-side reader for a rare event. **Revisit when:** the failure-email alarm proves too noisy from legitimate template churn (then switch to the flag-and-report path).
- **Out of scope, recorded as a future option:** a volume/size-drop canary (alert when a sender's daily volume or mean `CleanText` size drops vs a 7-day average — v3 §9), which would catch a footer-grew-instead-of-vanished failure the marker miss can't.
- Log lines (per-email for mapped senders + per-run rollup, real and DRY_RUN alike): [OPERATIONS — Collector: per-sender footer cutoff](OPERATIONS.md#collector-per-sender-footer-cutoff), with the **marker-miss runbook**.

## 5. Data model (Airtable)

- **Vacancies stays decisions-only** (applied/skipped/flagged) — no full vacancy inventory. **Rejected:** storing every parsed vacancy — the Airtable free-tier record cap; ~100 parsed vacancies/day would blow it in weeks.
- **Capacity budget — the free-plan record cap is per BASE, shared across all tables.** 1,000 records for RawEmails + Vacancies + everything else, crossed 2026-06-10 when a 3-week backlog sweep took the base to 1,018 (RawEmails 940 + Vacancies 77 + Vacancies_test 1). Enforcement observation from that incident: Airtable's over-limit notification arrived 11:13, record creation was still succeeding ≥5 hours later — treat enforcement timing as **unspecified**, never as headroom (see [KNOWN_ISSUES §6](KNOWN_ISSUES.md)). Allocation: RawEmails ≤ 700 (purge high-water), leaving ≥ 300 for Vacancies growth and slack. A second free-plan constraint: the **API-call budget** of 1,000 calls/workspace/month (verified against Airtable's docs 2026-06-10) — batch everything; one-call-per-record patterns are unaffordable.
- **RawEmails is a transient queue, kept inside its share by the purge job** (`purgeRawEmails`, same script file, own nightly trigger — time: §7): when the record count exceeds `PURGE_HIGH_WATER` (700, runtime-tunable), delete the **oldest** eligible rows down to `PURGE_LOW_WATER` (500, tunable); eligible = `Status='Processed'` AND `CollectedAt` older than `PURGE_MIN_AGE_DAYS` (2), enforced **server-side** by the `filterByFormula` so `Status='New'` rows structurally cannot be deleted — emergency purges of unprocessed rows stay a manual/owner action. Capacity-triggered with an age floor, not age-triggered (supersedes the earlier "purge after ~7 days" sketch): the trigger is the budget, the age floor is the safety margin. Starvation (over high-water, 0 eligible — the *normal* state pre-M6.2) logs a warning; at `PURGE_EMERGENCY` (950) with 0 eligible the run throws so the failure email fires before writes get blocked. **Rejected:** CI-cron janitor — violates §7's "no CI-triggered execution, ever"; Airtable Automations — no native delete-record action, "Run a script" needs Team plan, and a UI-pasted script is unversioned. **Revisit when:** the M6.2 cutover starts producing `Processed` rows at volume, or the purge hits starvation near the cap.
- **Schema-as-code, additive-only.** `airtable/schema.json` + an idempotent apply script in CI (Meta API). The API cannot delete tables/fields or change field types — removals stay manual. Caveat: matching is by field **name** until ID-matching lands (`TODO.md` → Airtable), so a UI rename re-creates the field as a duplicate on the next run — treat `schema.json` as the authority on names; see [KNOWN_ISSUES §3](KNOWN_ISSUES.md).

## 6. Screening layer

- **Instructions are versioned.** `VERSION: x.y` header (from 1.0): MAJOR = breaking (intake, gates, output contract), MINOR = non-breaking. Claude echoes the loaded version in every batch report.
- **Instructions are externalized to the repo file (M6.1).** The canonical instructions are `instructions/Claude_project_instructions.md` — the single `VERSION`-ed source under version control. The claude.ai project field is reduced to a **bootstrap pointer** (`instructions/PROJECT_FIELD_STUB.md`, pasted in by the owner) that reads the repo file from the mounted **UK_DevOps** folder. The contract is **local-file-primary, fail-loud-if-not-attached, no network/cache/memory fallback**: with the folder unattached the run **halts** (no screening, no Gmail read, no Airtable write, no labelling) rather than screening on stale or absent instructions. *Why:* the field previously held the full instructions inline, so edits to the repo file never reached the live pipeline — two sources that silently drift; a single versioned source removes that, at the deliberate cost that an absent mount must stop the run. **Rejected:** a network-fetch pointer (fetch the file from GitHub at run start, as `instructions/ChatGPT-Project-Source-of-Truth.md` does for the image project) — it reintroduces an availability/staleness dependency and would *not* halt on a missing mount; and keeping the inline copy (the status quo — repo edits never reach the pipeline). That slice changed **only the loading mechanism**; the instructions body stayed at `VERSION: 1.2` (canonical-source status documented in OPERATIONS rather than as an in-file header, to leave the authoritative text byte-identical — owner's call, M6.1). The **§1/§9 intake rewrite and the `VERSION: 2.0` bump shipped in M6.2** (the cutover-contract bullet below); the §6a link-resolution addendum is M6.3. Activation runbook + ordered owner checklist: [OPERATIONS](OPERATIONS.md#instructions-loading).
- **Cross-source vacancy identity rule** (shipped at the M6.2 cutover as instructions §5a): same title-pattern + rate band + location + stack via a *different* recruiter = same underlying vacancy. Keep one record, append "also via \<recruiter\> at \<rate\>" to Notes, never apply through a second channel once an application is in flight; prefer better terms / direct posting before first application; uncertain identity → flag, don't auto-merge. Rationale: duplicate agency submissions can disqualify the candidate.
- **M6.2 cutover contract (shipped).** Intake **is** RawEmails: read `Status=New` → screen → flip to `Processed` (instructions §1/§9 at `VERSION: 2.0`); the Gmail connector is demoted to a **discrepancy canary only** (0 New rows but unread Gmail mail = collector-failure alert, not a quiet day). There is **no Gmail-direct screening fallback** — an Airtable outage **alerts and stops** (nothing screened/marked/persisted; the next run catches up on recovery, since the collector's writes also fail during the outage so no mail is lost). This supersedes the earlier "fallback + canary" wording (Codex F1 [P2], owner decision 2026-06-16): post-2.0 the done-marker is the `Status` flip, so a Gmail-direct fallback that can't flip `Status` would re-screen on recovery — a dual-path design must hold the dedup invariant in *both* paths, so the second path was removed rather than reconciled. The Make.com scenario and the GAS collector stay live **in parallel as the safety net** for the first 2.0 runs. ## 8. Slice dispatch — Architect → Implementer handoff (tooling)

> **PARKED as of 2026-06-14.** The *cloud-dispatch automation* below (`dispatch-slice.sh` + the launchd watcher) is archived under `scripts/slice-passing-parked/`, not in active use: a CLI-launched `claude --remote` can't push in this environment (HTTP 401). The kept flow is the human-run `/run-slice`. Findings, the blocker, and revival steps: [slice-passing-parked/README](../scripts/slice-passing-parked/README.md). The decisions are retained below as the design record.

- **The Architect (Claude Cowork) hands each slice to the Implementer (Claude Code) as a GUI-visible session.** The Architect stages a slice under the gitignored `issue-drafts/<slice>/` (the spec `slice-prompt-<slice>.md` plus any fixtures); `scripts/slice-passing-parked/dispatch-slice.sh <slice>` cuts a branch off `origin/main` (prefix = work type, declared in the spec), force-stages that folder onto it, pushes, and launches the session with a working-agreement preamble (`scripts/slice-passing-parked/slice-agent-preamble.md`) prepended to the spec. It never dispatches from `main` and never merges. Mechanics, limitations, and the prompt template: [SLICE_DISPATCH](../scripts/slice-passing-parked/SLICE_DISPATCH.md), [SLICE_PROMPT_TEMPLATE](SLICE_PROMPT_TEMPLATE.md).
- **Decided: `claude --remote` (cloud), not `claude -p` (local headless).** A `claude -p` session is invisible in the desktop GUI — no Code-tab card, nothing on claude.ai/code or iOS — so the owner can't watch the slice run; `claude --remote` cloud sessions are GUI-visible and stream progress. They also bill against the normal subscription rate limits with no separate compute charge, unlike `claude -p` (which draws the separate Agent SDK credit — see [SLICE_DISPATCH](../scripts/slice-passing-parked/SLICE_DISPATCH.md#billing)). **Rejected `claude -p` for this handoff;** revisit only if a non-interactive/CI use case (no human watching) appears, and then add it behind a flag rather than replacing `--remote`.
- **Decided: Pattern A fixture staging (force-commit the staging dir), not prompt-inlining.** A cloud session runs in a fresh VM that clones the repo from GitHub and can't read local files, and `issue-drafts/` is gitignored — so the dispatcher force-commits `issue-drafts/<slice>/` onto the feature branch before pushing, so the clone carries the spec *and* fixtures. Inlining fixture *contents* into the prompt was rejected: byte-exactness (LF, the U+FFFD head quirk, per-recipient token redaction) doesn't survive a prompt string — let git carry the bytes. The Implementer then moves fixtures into their committed home and removes the staging dir, so after a squash-merge `main` shows only the fixtures in their home, never `issue-drafts/`. Fixture-heavy slices can instead run a **local** Code-tab session that reads `issue-drafts/<slice>/` straight off disk (no force-stage).
- **The launchd watcher ships opt-in and disabled.** `scripts/slice-passing-parked/watch-slices.sh` + `scripts/slice-passing-parked/com.kathelix.slice-dispatch.plist` are a GO-sentinel poll loop for hands-off auto-dispatch; they are **not** `launchctl load`-ed by default. Enable per the `watch-slices.sh` header (the canonical launchctl runbook) only with eyes on it, and point it at a dedicated worktree so auto-branching can't collide with manual work.
