## Collector (`apps-script/gmail-collector.gs`)

- [ ] **Fetch via label store instead of search index.** The `q=`-based listing (inherited from Make) reads Gmail's search index, which silently skips unindexed messages — observed 2026-06-07 with securityclearedjobs.com emails: visible in the Gmail UI, invisible to every API query (`from:`, `subject:`, `in:anywhere`). Switch to label-store listing (`getUserLabelByName('job-vacancies')` / `labelIds`-based) to make such orphans structurally impossible.
- [x] **`make-failed` on a repeatedly-transient write / set `make-processing`** — done (the cap) + decided (no `make-processing`). A record whose *own* Airtable write fails **transiently** (`429`/`5xx`/transport) on every run is now capped: a cross-run **Script Properties strike counter** (`wretry:<messageId>`) `make-failed`s it after `MAX_TRANSIENT_WRITE_RETRIES` (default 5, runtime-tunable, bounds `[1,100]`) **consecutive** strikes, so it stops retrying forever and failing the run every run; the counter clears on any successful upsert or on quarantine. Record-specificity is **sticky** (Codex F1 fix, Option 1): the **first** strike needs a same-run healthy sibling, then the counter carries the proof so the record keeps striking even once its siblings are collected and it is alone (else the cap is unreachable). A **fresh** never-struck message in a systemic outage never strikes (mass-quarantine guard, holds for the bulk of the queue); two accepted residuals (struck-then-outage, solo-from-start) are documented in `docs/TECH_DESIGN.md` §2 / `docs/OPERATIONS.md`. `DRY_RUN` does no strike/quarantine/Script-Properties write. **`make-processing` is deliberately NOT set** (as in Make): single-flight + the idempotent write-then-label already give crash-safety, so a mid-flight marker would add persistent state for no benefit (owner decision 2026-06-15). The cross-run counter stays **rejected for the deterministic case** (slower, doesn't unblock siblings — handled within-run by the per-record isolation). See `docs/TECH_DESIGN.md` §2 (repeatedly-transient write cap) and `docs/OPERATIONS.md` (the `wretry:`/`MAX_TRANSIENT_WRITE_RETRIES` runbook).
