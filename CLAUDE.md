## Agent roles & slice workflow

Work moves in **slices**, one role per agent:

- **Architect — Claude Cowork.** Owns the whole-system view and the roadmap; cuts the next slice and hands it to the Implementer as a self-contained **markdown prompt** (scope, old→new behaviour to preserve, acceptance criteria/tests, guardrails, out-of-scope). Does **not** push, open PRs, or merge.
- **Implementer — Claude Code.** Takes one prompt, works it on its own branch + PR, then resolves Codex's review. After the owner approves, performs the (squash) merge, watches CI to green, and reconciles the local checkout to `main` (per *GitHub PR hygiene*).
- **Reviewer — Codex.** Reviews each PR. **Architectural** findings — intake contract, screening gates, output shape — are referred back to the Architect before they're resolved; code-quality findings the Implementer fixes directly. This holds **even when the conflict originates in the slice prompt's own wording** rather than the Implementer's code — refer it, don't quietly reconcile it in place (PR #48 F1: the prompt's "resolving the origin can confirm the gate" note collided with the retained "an aggregator's own page is not sufficient" rule; it went to the Architect, who scoped origin content to discovery-evidence-only). **The refer-vs-fix line is whether the resolution needs a design *decision* or is *forced* by existing shipped code:** a finding framed as a "contract"/"design" issue is still a **code-quality fix the Implementer makes directly** when resolving it requires **no choice** — the code already determines the answer and the docs merely *lag* it (a description that under-describes unchanged behaviour), and it touches none of the intake/gate/output-shape surfaces — but **state that classification explicitly in the reply** so the Architect can object. Refer only when the fix requires *choosing* between behaviours or rules, as PR #48 F1 did (PR #49 F1: "one marker per template" → "candidate footer starts" was forced by the existing earliest-valid-cut code and changed no behaviour, so it was resolved directly with the classification flagged; Codex's re-review accepted it without Architect escalation).
- **Owner — Ivan.** Final review/approval on every PR; reports "slice done" back to the Architect, who proposes the next slice.

The merge *decision* remains the owner's — Claude Code only executes a merge the owner has approved on that PR; nothing auto-merges. This file is canonical; keep the same section in `AGENTS.md` so Codex reads identical rules.

**Every message meant for another agent is one self-contained fenced Markdown block — Ivan is the human relay.** The three agents run in separate tools (Cowork, Claude Code, Codex) with no direct channel; Ivan copy-pastes between them. So whenever you dispatch something for another agent to act on — the Architect's slice prompt, the Reviewer's findings or architectural referrals back to the Architect, the Implementer's reply resolving a review, a relayed "slice done" report — emit it as a **single fenced Markdown code block** Ivan can copy whole into the next tool (generalising the Architect's *self-contained markdown prompt* above to every handoff). Open the block with a routing line naming sender→recipient and the slice/PR it concerns (e.g. `_From Claude Code (Implementer) → Codex · PR #46_`) so the receiving agent — which has none of this session's context — can orient. Keep any narration addressed to Ivan **outside** the block so the copy boundary is unambiguous; and when the message itself contains a fenced code block, lengthen the outer fence to four or more backticks so an inner three-backtick fence can't close it early. File-staged slice inputs (next paragraph) are the complementary path for slice *inputs*; this rule governs anything relayed as a chat message.

**Slice staging is the Implementer's to clean.** The Architect may stage a slice's inputs — the prompt, fixtures, a provenance/redaction README — as untracked files under `issue-drafts/<slice>/` (gitignored scratch, never on `main`, never committed by the Architect). The Implementer *consumes* that staging as part of the slice: **move** (not copy) fixtures into their committed home, fold the provenance into the PR body, then remove the `issue-drafts/<slice>/` dir so the working tree ends clean. Untracked staging survives `checkout`/`pull`/`merge` and shows in no diff, so nothing else clears it — leaving it is how scratch silently accumulates (PR #18). The spec itself lives at `issue-drafts/<slice>/slice-prompt-<slice>.md`; run bare, `/run-slice` lists the staged `slice-prompt-*.md` files and asks which to implement (auto-running only when a single one is staged). The **cloud-dispatch** variant that *force-commits* the staging dir onto a feature branch first — so a cloud clone can read the otherwise-gitignored files — is **parked**; see `scripts/slice-passing-parked/README.md`.

## Documentation & decisions

- **Read the project runbook before reconstructing a recurring procedure** — operational how-tos live in `docs/OPERATIONS.md` (e.g. adding/confirming a `FOOTER_MARKERS` entry → *Screening: footer-freshness alert*; the ≥2-sample byte-form confirmation is a RawEmails **history pull**, not just the day's sample), with design rationale in `docs/TECH_DESIGN.md`.
- If the project keeps design docs, treat them as the source of truth and keep code from getting ahead of them: decide, then spec, then implement. A common split is a stable high-level design (the "why"), a concrete technical spec (the "what" — versions, names, constraints), and a backlog (implementation status). Update the spec before writing the code it describes.
- Mark a backlog or spec item done in the **same commit** that implements it — never let recorded status drift from reality. **How** you mark it done is set by the backlog's own contract: tick `[x]` where the file keeps a closed-item trail, but where the file declares it tracks **only open work** (shipped items live in the permanent docs), *done means **removing** the item* — a ticked item left in an open-only backlog is itself status drift, the file contradicting its own header. Read the header; don't tick by habit, even when a slice prompt explicitly says to (PR #36 [P3]: the prompt said tick `[x]`, but `TODO.md`'s open-only contract from PR #35 meant the shipped item should be **removed**, its record kept in `KNOWN_ISSUES §3` + `TECH_DESIGN §5`; Codex flagged the contradiction).
- Record decisions that are architectural, security-relevant, hard to reverse, or likely to need explaining later in `docs/TECH_DESIGN.md`, in the section they belong to — including rejected alternatives and revisit triggers, so settled questions aren't re-litigated. Update it in the same PR as the change it describes. It is a living document: edit in place; history lives in git.
- **A design-doc record must not assert or mandate behaviour the executable contract doesn't have — an "already in-contract" claim is a *verify-before-claim* hypothesis.** When you record a decision in `TECH_DESIGN`, a clause saying the system "already" does X (or *should* do X) is true only if X is actually encoded in the instructions/code — trace it to the specific §/line before committing it, **even when transcribing an Architect's exact wording** (the Implementer still verifies). A record that invents an un-encoded rule contradicts the real contract and reads as settled fact. This is the mirror of *keep code from getting ahead of the docs*: keep the **record** from getting ahead of the **contract** (PR #48 F6: a ratified revisit-trigger note called it the in-contract "drill floor" that recruiter-only roles are "kept a Flag, never auto-upgraded" — but that mandate was nowhere in the instructions and contradicted §6a's gates-hold → >75% upgrade; it was removed and the case left to the existing §3/§6a/Block-2 rules).
- When deleting or consolidating prose, sweep for dependents in the same change: re-home work items embedded in the prose (obligations not formatted as tasks) into the backlog, and reword backlog items contingent on anything the change supersedes — a sweep keyed to task markers alone misses both.
- **The deletion sweep has an *inbound-pointer* direction a behaviour-keyed grep misses: when you *remove* an item other docs point **at**, those inbound pointers dangle.** A grep for the *behaviour's words* won't find them — a `(see X)` cross-ref, a "tracked in X", or a link to the item's anchor is keyed to the **container's name/anchor**, not the behaviour. So when removing content, also grep every **inbound reference to the container** (the filename, the heading anchor, the item's title) and re-resolve each hit against the post-removal target: **drop** the pointer if the fact now lives inline at the pointer site, **re-home** it as an open item if it's still live work, or **re-point** it to the fact's real permanent home. And **act on every hit the sweep returns** — running the right grep but fixing only the obvious subset is how a dangling pointer survives (PR #35 F1: removing the ticked `TODO.md` items left `(see TODO)` Make-decommission pointers in `TECH_DESIGN` §6 + `OPERATIONS`, and a "no GC sweep yet — `TODO.md`" note aimed at a backlog item that never existed; the upfront `grep -rn TODO docs/` *showed* all three, but only the README hits were fixed — Codex caught the rest).
- The deletion sweep has an **additive mirror image**: when a change adds to something a doc enumerates or quantifies, re-read the enclosing section for claims the addition falsifies — counts ("~150 lines of pure functions"), totals, "only/both/all" wording — not just for mentions of the thing touched. In rationale prose prefer magnitudes and revisit criteria over precise volatile numbers; a pinned count goes stale on the next slice (PR #16). A **bounded exception** added to a previously-absolute rule also falsifies the rule's *absolute phrasings* — `never`/`nothing`/`always`/`only`/`any` — which a grep keyed to the *behaviour's* words misses, because they restate the invariant in different words and usually sit in **summary** prose far from the feature you touched (PR #27 F2: the cap sections documented the new residual, but the broad "a systemic outage quarantines **nothing**" lines in the isolation path and the operator runbook still over-claimed; Codex caught it).
- **The doc sweep spans every surface that *describes the changed behaviour* — the operator runbook (`docs/OPERATIONS.md`) most of all, not just the design doc, spec, and backlog.** A change to a failure-handling path falsifies the same claims in the routine procedures and troubleshooting tables an on-call human reads under pressure, where a stale triage row is higher-stakes than a stale design note. Grep the behaviour's *old* description (e.g. "whole sub-batch uncollected") across **all** of `docs/` in the same change, not just the spec (PR #26 F1: the transient-isolation slice swept `TECH_DESIGN`/`TODO`/test docs but left `OPERATIONS.md` still telling operators a transient failure leaves the whole sub-batch pending; Codex caught it).
- **A behaviour change's doc-sweep must re-read the *procedure*, not just grep the new keywords — a runbook step is a *descriptive container* that can describe the changed behaviour without naming it.** When you **add** a behaviour (a new rule, mode, or distinction), the runbook procedure that *performs* it needs syncing even though your new keywords never appeared there before — so a `git grep` for those keywords finds nothing and a keyword-keyed sweep skips it. Re-read each procedure that *does* the thing, not only the lines that *mention* it (PR #48 F4: the §6a origin-first / no-paywall rules were added, but the `OPERATIONS.md` Chrome-pass procedure — which carried none of those keywords — went unsynced until a re-read of the steps themselves caught it; the keyword grep had come up empty).

- **Removal has a historical-record trap a behaviour-keyed sweep misses: a *dated* incident record is a point-in-time snapshot, not current state.** When an entity is retired and its mentions are scrubbed, don't blindly delete it from a dated record — that can **falsify the dated fact** and **break a reconciling sum** (a breakdown that must total a stated figure), and the same number is often **mirrored in a cross-referenced doc** that has to change in lockstep. An owner may override a "leave verbatim" note, but then keep the arithmetic honest (adjust the total) and apply the edit to *every* linked copy in the same change (PR #30: dropping `Vacancies_test 1` from the 2026-06-10 capacity record forced `1,018 → 1,017` in **both** `docs/TECH_DESIGN.md` §5 and the cross-referenced `docs/KNOWN_ISSUES.md` §6).
- **Shipping a milestone falsifies two backlog/runbook classes a behaviour-keyed sweep misses: items that assume it is *not yet* shipped, and items the milestone *completes* elsewhere in the list.** Grepping the changed behaviour's words finds the prose that *describes* it, but not (a) a runbook remedy phrased "accelerate/await \<milestone\>" that goes stale the moment it ships, or (b) an open backlog task the slice actually finished in another section. So also grep the **milestone name and its state-words** (`M6.2`, `pre-cutover`, `once parity is confirmed`, the task's own title) across the backlog **and** the runbook, and tick every item the slice closes done in the same commit (PR #29 F2/F3: the purge capacity-emergency row still said "accelerate the M6.2 cutover (nothing is `Processed` pre-cutover)", and the screening-side-canary `TODO` the slice shipped was still open; Codex caught both).
- **A version/identifier *bump* falsifies pinned copies of the *old* value — the additive-mirror sweep covers identifiers, not just counts.** After bumping a `VERSION` / schema id / similar token, grep the **old** string across `docs/` and de-pin or fix every **current-state** reference (prefer pointing at the canonical source over re-pinning the value — state tunable facts once); leave **dated / milestone** records of the old value verbatim (point-in-time history, per the historical-record trap above). A behaviour-keyed sweep misses these because the stale token is the *identifier*, not the behaviour's words (PR #38 [P3]: bumping the screening instructions `VERSION 2.1 → 2.2` left a current-state `VERSION: 2.1` pin in `OPERATIONS.md` that the runbook's own version-echo check would read as stale; the M6.x milestone records of `2.1`/`2.0` correctly stayed).
- **A new metric/gauge's operator-facing description must not claim diagnostic power it doesn't have — write it *correlate, don't conclude*.** An aggregate/volume gauge measures a *quantity*, not a *cause*: it cannot attribute *why* the number moved, and in particular it can neither prove a fault **nor prove a fault's absence**. So a runbook line for a new metric must not say a movement "tracks X, **not a fault**" (nor "means everything's fine") — name the **companion signals** to cross-check (the related count, the relevant alerts, any mode/fallback label that changed the *measurement method* rather than the underlying quantity) and state plainly it's a prompt to correlate, not a verdict. Kin to the *silent-success-hides-a-real-failure* family but on the **doc** surface: an over-claimed metric reads as reassurance it can't give (PR #43 F1: the §8 proxy-token gauge's `OPERATIONS` line claimed a jump/drop "tracks intake volume, not a fault" when a partial collector/index miss or footer drift also moves it and the `chars/4` fallback changes the method; Codex flagged it).
- **An *"exactly / same as / this is"* equivalence to another doc's concept is a claim about *that concept's* semantics — verify its direction/trigger/scope before asserting it.** Topically adjacent is not equivalent: a cross-reference that equates your example with a named signal elsewhere must match its **polarity and trigger**, not merely its subject. Re-read the referenced item's actual definition (as for any *pattern-claim*) before writing "this is exactly X" (PR #43 T1: an `OPERATIONS` line equated an un-cut footer *inflating* `CleanText` with the `TECH_DESIGN` §4 volume/size-**drop** canary, which alerts on anomalous *decreases* — same topic, opposite direction; the fix mapped each mechanism to its own signal, inflation→today's marker-miss/footer-freshness alerts, deflation→the future §4 canary; Codex caught it).
- **An abstraction documented from a single instance is over-fitted to it — the *second* instance is the test: generalize the framing, don't force-fit or exception it.** A contract/feature described when it had **one** example (an array "for senders with multiple templates", a mode, a rule) often states a scope narrower than the **code** actually implements — the lone example never exercised the general case. When you map a *second* instance onto it, check whether the stated scope truly covers your instance; if your instance fits the **code** but not the **documented framing**, the framing was the over-fit — **generalize it to what the code already does**, don't relabel your instance an exception or force it into the narrow wording. The *first* instance has often already violated the narrow framing unnoticed (PR #49: the marker-array was documented "one marker per template", but the NIJobs digest — its only prior user — already carried two markers *within one template*, and `footerCutIndexMulti_` had always cut at the earliest valid *candidate*; milkround, digest-only with two candidate cut points, made the contradiction undeniable — generalized to "candidate footer starts" across all four surfaces, no behaviour change). **The same array also covers a sender whose templates *order the same footer elements differently*** (not only different element sets): if one marker cleanly cuts template A (e.g. postal-first) but in template B (action-first) leaves a per-recipient action endpoint *before* it, that is **not** a residual to ship or a case to defer — add a **second confirm-twice marker** for B's earliest footer element and let earliest-cut-wins pick per template (the marker that wins in A is the non-winning *later* candidate in B, so neither over-cuts). A referral's *"this second marker is n=1"* premise is itself a hypothesis — check the stored bytes first (PR #52 F1: nexxt's action-first `jfw@` template left its `/optout` before the postal marker; the fix was a 2-marker array `[postal, optout-intro]`, and the optout-intro proved byte-identical across all four captured messages, not `jfw@`-only).

## Branching & commits

- Feature work goes on a branch; open a PR to the main branch. Use descriptive names (`feature/`, `fix/`, `chore/`).
- **Branch = work type, declared by the Architect in the slice spec.** A `<!-- branch: chore/x -->` comment names the branch outright; `<!-- type: fix -->` shorthand → `fix/<slice>`. The prefix is a naming **convention** (`feature/`, `fix/`, `chore/`, `docs/`, `refactor/`), defaulting to `feature/<slice>` when unset — not an enforced allow-list, so read the resolved name back before pushing (a typo'd prefix is a valid branch name, not an error). The Implementer applies this when naming the branch for `/run-slice`. *(The parked `dispatch-slice.sh` automated the resolution — CLI arg > `branch:` > `type:` > default, with a `--dry-run` preview; see `scripts/slice-passing-parked/README.md`.)*
- When creating a branch, branch explicitly from the intended base (`git checkout -b <name> origin/main`) and verify it before committing (`git log -1`, `git merge-base --is-ancestor`) — don't trust the session's start-of-conversation git snapshot, which can be stale. Run parallel agent sessions in separate git worktrees so their branches can't collide.
- Don't start feature work on the main branch — not even docs-only edits. The only exception is a tiny mechanical change the owner has explicitly approved for direct-to-main.
- Commit subjects: short, imperative, no trailing period (e.g. `Add upload retry, persist draft state`).
- When you address review feedback or follow-ups on a branch, make a **new commit** rather than amending the original — separate commits let a reviewer reconstruct which change answered which round. Amend only for trivial typo cleanup, and say so first.
- Keep `Co-Authored-By:` trailers accurate to the actual model or agent that did the work.

## Verify before you claim

The single most valuable habit: never report something you have not just verified. Cheap to check, expensive to get wrong.

- **Git mutations:** verify against live state, and never announce a commit or push in the same step that performs it. Afterward, re-read the truth — compare `HEAD` to `origin/<branch>` to confirm a push landed, and count with `git rev-list --count` before stating a number. Never quote a commit SHA you have not just read back.
- **Silent no-op edits:** an `Edit`, `git mv`, or `sed` that targets the wrong file or a misremembered string fails _silently_ — it changes nothing. So author every commit message, PR body, or follow-up task from the **verified post-change diff** (`git grep` to prove no stale references remain; read back `git show --name-status`), never from what you intended to change. Write each edit's match text from a fresh read of the file, not from memory.
- **Negative claims** ("this flag/API/rule doesn't exist, remove it") are higher-stakes than positive ones, because the next step is usually deletion. Prove the absence empirically — enumerate from the installed package, run the tool with it enabled, grep the source — before posting. And a capability's **absence is environment-scoped**: "no tool/API/path exists" is a claim about *your* connector/toolset, not the project — a peer agent's environment may expose what yours can't. Prove what *you* lack by enumerating your own tools, then **scope the claim** ("not in my connector"); never assert it project-wide from your-toolset-only evidence (PR #50 F1: I wrote `get_thread` is "the only capture path" / "no byte-preserving RFC822 export exists", but Codex's connector had the real raw export `read_email(include_raw_mime)` — the absolute was false; scoped to Claude Code's connector and re-stated the true remaining blocker).
- **Pattern claims** ("this regex/grep/guard misses a case") need a scratch reproducer: build a small file with the suspected variants, run the existing pattern against it, and show the result — before and after the fix.
- **A matcher is a hypothesis — including the one a task or acceptance criterion hands you; prove it catches the variants before trusting an empty or "all-clear" result.** `git grep`'s default ERE/BRE does **not** honour `\b` as a word boundary, so `git grep -E "\bWord\b"` matches **nothing** for the bare word. An acceptance criterion's own `git grep -nE "Foo\.com|\bFoo\b"` quietly matched only `Foo.com` and hid every bare-`Foo` mention — a sweep keyed to it would have missed them. Reproduce the matcher on a scratch file of the suspected variants (as for any *Pattern claim* above) and use `-P` (PCRE) or `-w` for word boundaries (PR #33: the bare-`Make` `(Make parity)` mentions were invisible to the ERE grep; `-P`/`-w` caught them).
- **A mechanical token-rename isn't done when `git grep` goes clean — the strip can break the prose it edits, and the acceptance pattern under-counts.** A global find-replace silently leaves ungrammatical (`may make-failed` → `may failed`) or meaning-shifted (`never make-fails` → `never fails`, now read as "the op didn't fail", not "never labelled failed") text wherever the old token doubled as a coined verb or load-bearing word. Two habits, both extending *matcher-is-a-hypothesis* above: (1) **sweep every inflection and sibling token, not just the acceptance pattern** — the `make-(collected|failed|processing)` criterion silently passed over `make-collects`/`make-fails`/`make-faileds` and a rejected `make-stuck`, all caught only by a broader `make-[a-z]+`; (2) after the strip **re-read the hits** — or grep the new bare token after a modal/aux (`may|to|can …`) and before an object (`… a|the|nothing`) — and repair to an explicit phrase (`mark X failed`) (PR #34).
- **Numbers measured by a prototype are corridors, not goldens.** When a spec ships values measured by scratch/research code (counts, bytes saved, percentages), accept the implementation through a corridor (floors/ranges that absorb honest variance), then re-measure with the shipped code and pin **its** exact values in the committed test — the corridor catches a broken port now, the exact pins catch silent regressions later. Never copy prototype numbers into goldens without re-measuring (PR #16, where the slice prompt pinned this pattern; it generalizes to every research-backed slice).
- **To check whether a transform *fired*, probe its INPUT — never a stored *post-transform* value.** A stored/processed artifact already reflects the very transform you are testing for, so probing it for the transform's *trigger* returns a **false negative by construction**. Read the pre-transform artifact (re-run the pipeline up to that stage), or prove the transform ran via an *output* invariant (e.g. `stored length == cutIndex(marker)`), not by hunting the trigger in the output. This is a higher-order sibling of *test-harness semantics* and *a staged draft's self-claims*: the diagnosis is run against the wrong artifact, and the whole slice premise rides on it (UK_DevOps PR #44: a NIJobs digest footer was diagnosed a fail-loud `miss` because `lastIndexOf('<the cut marker>')` was run against the **stored `CleanText`** — which is *already cut at that marker*, so the marker sits in the discarded tail and the probe is `−1` by construction; the live behaviour was a clean `hit` that merely left an *earlier* residual. The slice's miss/alarm-fix premise was false; it was paused and re-spec'd to residual-removal — the fix and capability were still correct). **The blind spot recurs on the *consumer* side, not just at diagnosis: a downstream stage that reads only the stored output cannot recover an unobservable upstream distinction, and the output's *residue* is not a valid proxy for it.** The scheduled screening run reads `CleanText` and does **not** retain the collector's per-message `hit`/`miss`, so a §3 rule **must not** be based on it or on a `CleanText`-shape proxy. When an action's safety hinges on a distinction you can't observe, take the action **safe under every branch** (UK_DevOps PR #45: a §3 rule split *drift* from *residual* by "how much footer remains", but a `hit` can keep a pre-marker endpoint and a `miss` can leave a short fragment — both defeat it; the contract collapsed to **append-always for a matched key**, since append ⊃ replace: a stale appended marker is a harmless `−1`, while deleting a still-live marker breaks the other template).
- **Reverting a temporary mutation:** to prove a guard is load-bearing, break it and watch a test flip — but first copy the file to a scratch path (`cp f /tmp/f.bak`) and restore from that. Never undo the mutation with `git checkout <file>` when the file has uncommitted work: it resets to `HEAD`, silently discarding _all_ your uncommitted changes in that file, not just the mutation. Re-run the suite after restoring to confirm you're back to green.
- **Test-harness semantics** are a hypothesis too: when a test loads code through a sandbox, separate realm, transpile, or mock layer (a `vm` context, a worker, an iframe), confirm the harness's own identity and equality behavior with one throwaway assertion before trusting the suite — across a realm boundary `instanceof` is `false` and strict deep-equality fails on the prototype check, so a green run can be asserting the wrong thing. Prefer primitive leaves, serialized forms, or `.source`/`.flags` over object identity.
- **A stubbed runtime hides runtime-only behavior.** When the harness *replaces* a platform primitive instead of running it (the `vm` loader stubs `Logger`/`Utilities`, so `Logger.log`'s `%s` is substituted JS-side), anything that only happens on the real platform is invisible to a green suite — live Apps Script routes `%s` through Java and prints an integer as `114.0`, which the Node harness cannot reproduce (PR #12). A unit test would pass identically before and after such a fix (false assurance), so verify on the real runtime instead. The agreed GAS loop: the agent hands over a minimal, side-effect-free throwaway script, the owner runs it in the editor and returns the log.
- **Guards around a tested predicate.** Unit-testing a predicate (`isOverBudget(now)`) does **not** test the `break`/branch that calls it — the guard can be deleted with the suite still green. When a fix wires a tested predicate into a new loop break or branch, add a test that fails if the break is removed (drive the loop with an injected clock/counter, then assert via a mutation check that deleting the break flips it), or mark that path manual-only in the test docs. A harness that exports **only the pure helpers** makes this gap *structural* — the wiring is unreachable from the suite, so the guard reads as covered while nothing tests it; export the entry point too and drive it through injected stubs, asserting the side-effect count (e.g. **zero** writes on the guard path), mutation-checked (PR #41 F2: deleting the empty-result `if`/throw from the entry point left all pure-helper tests green). And when a budget/timeout guard can skip a _later_ phase that flushes already-produced work, guarantee minimum forward progress (e.g. flush one batch) or document the starvation case — idempotent retries protect data integrity, not liveness.
- **A "it'll recur next run / nothing is lost" argument breaks at the commit/label boundary.** Before justifying a suppressed or deferred signal with "the next run picks it up," check whether the work it rides on is already committed and marked done (labelled, status-flipped, dequeued) — committed work never re-presents, so a signal attached to it will not recur. Carry the signal forward in the same failure/output rather than relying on recurrence (PR #17 F1: a footer-marker miss raised in a sub-batch that had already upserted + labelled was swallowed by a co-occurring upsert-failure throw; the fix folds the miss summary into that throw).
- **Live state and stated assumptions** are hypotheses until checked at the source. "Is this still in use, did that deploy land, is this quota hit" — read the live source (metrics, console, an API call) rather than assuming, including when the assumption came from the owner.
- **SDK / CLI behavior:** read the pinned or installed source (the dependency cache, the package's own files, `--help`), not memory or public docs, which may describe a different version. This applies at design time too — don't build a plan around a guessed capability.
- **Cross-platform shell:** utilities like `stat`, `date`, `sed -i`, `readlink -f`, and `mktemp` behave differently on BSD/macOS versus GNU/Linux, often both exiting `0` with different output. Branch on `uname` explicitly, or run on both before claiming green; a `bsd_form || gnu_form` fallback is a smell. Name the host in any verification table so "green" isn't mistaken for platform-portable.
- **Don't glue a non-ASCII character directly onto a `$var` expansion** (e.g. `$BRANCH…`) — macOS's bash 3.2 mis-scans the trailing multibyte byte into the parameter name, so `set -u` aborts with a garbled "unbound variable". Brace-delimit: `${BRANCH}…`. Locale-gated: it fires only in a UTF-8 locale (`C.UTF-8` included); a repro forced to `LC_ALL=C` shows nothing, so reproduce under `LC_ALL=en_US.UTF-8`.
- When tool or shell output looks stale, truncated, or self-inconsistent, stop and re-fetch from an authoritative source before acting — don't edit by a line number or post a value you're unsure of.
- **A staged draft's self-describing claims are unverified until you run them.** The slice rule is *move, don't author from scratch* — but moving a draft into place is not the same as trusting its prose. When a draft's docs or comments assert behaviour about its own code ("validates X", "fails fast on Y", "rejects Z"), treat it as a pattern-claim and reproduce it before committing — a draft that over-promises ships the over-promise. And a code-vs-docs mismatch has **two** valid fixes — tighten the code, or correct the docs; choose by which side states the intended contract, and when a documented owner preference decides it (e.g. value-guards over enumerated allow-lists), surface the choice rather than defaulting (PR #21 F1: staged `SLICE_DISPATCH.md` claimed a typo'd branch prefix "fails fast", but its sole guard `git check-ref-format` accepts any well-formed ref; Codex caught the mismatch — the owner kept the prefix list-free and the docs were corrected).
- **Runtime/harness behaviour is a hypothesis even when a guide or helper agent states it confidently — capture one real instance, and don't infer cause from correlation.** A docs-trained guide (or a purpose-built guide agent) returns *some* version's answer, not necessarily the installed one — confirm by capturing a real instance (instrument the actual hook/tool to dump its input, fire it once, read the bytes). And a state change that *follows* your action isn't proof it *caused* it: run a controlled probe isolating one variable before asserting "X is broken." (PR #22: a guide agent said a Bash hook's output was at `.tool_output.stdout`; the captured payload showed `.tool_response.stdout`, and the filter it blamed was actually fine — a clean probe overturned my first, wrong diagnosis.)
- **A hook that writes an app's own state file out-of-band to steer the *running* app is fighting the app's ownership — verify the live effect, and re-check for a native API before *maintaining* the workaround.** The app drives behaviour from its *in-memory* state and rewrites/compacts its files on its own schedule, so an out-of-band write (a) doesn't change the running app — it's re-read only on the next restart/resume — and (b) can be **dropped wholesale when the app rewrites the file** (e.g. compaction). A workaround's own "no native API exists" note also goes stale: re-read `--help`/the installed source, since the platform may since have added a first-class mechanism that supersedes the hack. (PR #37: `pr-session-name.sh` appended a `custom-title` record to the Claude Code session journal to rename a session; the record format was correct, yet it never retitled a *live* session — in-memory wins, effect only on resume — and this long session's **compaction dropped the append entirely** (0 records survived); meanwhile Claude Code had added native `pr-link` records + `claude --from-pr`, so the hook was **retired, not fixed**. Extends the PR #22 lessons on this same hook.)

## GitHub PR hygiene

- **PR body** captures: the original request or bug (preserve the user's error text or screenshot), root cause, what changed, how it was validated, manual-test status, and any review follow-ups. It is the durable validation/review summary, so when a review-follow-up commit changes a number it quotes (test count, byte delta, file count), update the body in the **same push** — a stale count reads as "not re-run" and is the additive-mirror of the doc-count sweep above (PR #20: Codex flagged a `146`→`148` test-count drift after the F1 follow-up commit). A quoted count usually appears in **more than one place** in the body (a Validation total *and* a per-file Changes line), so grep *every* occurrence, not just the first (PR #30 [P3]: the Validation total read 204 while a Changes bullet still said "18 offline cases"). This is not only about numbers — a follow-up that changes the *described contract/behaviour* (e.g. a fallback became alert-and-stop) must update the PR **title and body** in the same push too, since the body is the durable summary and a stale contract description contradicts the merged diff (PR #29 [P3]: after the F1 re-spec the body still described the removed Gmail-direct fallback; Codex flagged it).
- **Issue links:** use `Part of #N` or `Refs #N` for a slice of a larger issue; use `Closes #N` only on the PR that actually completes it. After merge, **verify the issue actually closed** — squash-merges sometimes record only a reference — and close it by hand if needed (housekeeping, not a merge decision).
- **Avoid auto-link collisions:** GitHub turns `#N` into a cross-reference, `@name` into a mention, and bare URLs into unfurls. Label review findings with distinct prefixes per round (`F1`, `F2`, then `T1`, `T2` for test follow-ups) so they don't read as issue links.
- For complex PR bodies, write the Markdown to a temp file and pass `--body-file` instead of escaping it into one shell argument. In the default `zsh`, don't name a variable `status` (it's reserved) — use `rc`.
- **Check every comment surface — there are three, and a shared account hides authorship.** PR feedback lives in (a) top-level **conversation** comments (the issue-comment stream), (b) **inline** review comments on lines, and (c) the **reviews** array. A bot or Codex review can land in *any* — Codex posted its entire review as a top-level conversation comment while an inline/`reviews`-array check came up empty — so query all three (`gh pr view --json comments`, the `pulls/{n}/comments` inline endpoint, **and** `--json reviews`) before concluding a PR has no findings. On a **shared account** (Codex posts as the owner), distinguish authorship by the body **marker** (`_Posted by Codex_` vs `_Posted by Claude Code_`), **not** the author login — an author-based filter false-positives on your own replies (PR #30).
- **Which checks run isn't fully knowable from `.github/workflows/`.** GitHub-managed checks (CodeQL default/dynamic setup and the like) run on a PR with no repo workflow file and no `paths:` filter — so a "docs-only, nothing triggers" inference from the workflow files alone can be wrong. Confirm the real check set with `gh pr checks <n>` on the live PR, and read each result, before claiming what runs or that the PR is green.
- **Merge is the owner's decision — never the agent's.** Get the PR merge-ready (CI green, findings resolved, body accurate) and hand it over. No `--auto`, no poll-and-merge, no "it was green so I merged it." A "merge it" instruction is per-PR, not standing. When asked to merge, default to squash. After a merge: switch to the main branch, `pull --ff-only`, delete the old branch, **then verify the post-merge `main` CI *for the merge commit* before reporting the slice done.** The squash is a *new* commit on `main` whose `push`-triggered workflows never ran on the PR — the **GAS deploy runs only on push to `main`**, never on the PR — so a green PR does **not** imply a green `main` (a transient deploy failure leaves the live system **un-deployed** while the PR shows merged-and-green and the test job stays green). Procedure: **(1)** capture the squash **merge OID** (`gh pr view <n> --json mergeCommit -q .mergeCommit.oid`) and query **that commit**, not just the branch — `gh run list --branch main --commit <oid> --json name,conclusion,headSha` — because bare `--branch main` can surface a *different, later* merge's runs and falsely certify this one. **(2)** Derive the **expected** workflows from the merge's **changed paths** (each is `paths:`-filtered): require **Deploy GAS** green only when the merge touched one of its triggers (`apps-script/**`, `.clasp.json`, `.claspignore`, `.nvmrc`, `deploy-gas.yml`); otherwise record it **“not scheduled”** — a docs-/instructions-only merge runs no deploy, and you must **not** borrow an unrelated earlier green deploy as proof. The Deploy-GAS step **self-retries once** on a transient error, so a failure that *survives* the retry is likely real — investigate it, don't just re-run. (The retry's sleep is a small window in which an older run could overwrite a newer deploy; this is an **accepted risk** for a solo, low-volume project — mitigated by this very check + the CI-failure email, revisit if deploys become concurrent — see the `deploy-gas.yml` comment, PR #53.) When a deploy **is** scheduled, the slice is not done until it is green **for the merge commit** (PR #51: a transient Deploy-GAS "Premature close" on `main` went unnoticed, so talent.com was not actually live until PR #52's merge re-deployed it). This rule also shapes the permission allowlist: `gh pr merge` is never added to the no-prompt allowlist, even when other `git`/`gh` mutations are (PR #18).
- **Attribution:** every PR comment or description an AI agent posts starts with a short italic line naming the agent, on its own line at the very top — e.g. `_Posted by Claude Code_`. On a shared account this is the only way a later reader tells AI-authored comments from the owner's.

## PR review discipline

- Open every review with a one-line "is this PR right-sized for what it claims to do?" check; flag scope creep before drilling into findings.
- Gate by severity with explicit "block on" versus "nice to have." Cluster trivia under one "Nits" heading rather than numbering each; if an item needs hedging ("borderline," "arguably"), drop it instead.
- **Review deep, not just the diff.** Cross-reference the change against the unmodified files it touches — config, state, validation, callers — to catch edge cases and logical breaks the diff alone hides.
- **The PR thread is the durable record; chat is only transport.** Post every structured finding, architectural referral, re-review result, and no-findings / merge-ready conclusion to the **PR first**, each carrying the agent's attribution marker (*Attribution*, GitHub PR hygiene); then emit the self-contained fenced relay carrying **only the outcome/action plus the PR-comment link** — never a duplicate of the full details. A finding sent only as a chat relay block never reaches the durable record the next reviewer reads (PR #48: F1–F4 were relayed but initially absent from the PR thread).
- **Route the relay to the agent who must act — not to the human relay.** Architectural decisions → Claude Cowork (Architect); code-quality fixes, re-review outcomes, and no-findings / merge-ready results → Claude Code (Implementer). Ivan is the **relay, not the recipient**: name the acting agent in the routing line and keep any Ivan-narration outside the fenced block (PR #48: a no-findings result was addressed to Ivan rather than the Implementer who acts on it).
- **Verify the durable write against live state — a returned URL is not proof.** After posting, re-fetch the comment body/URL **and** the current PR head; if the head advanced while you were posting, re-review *that* head before any status or no-findings claim. A relayed review can quote a **superseded snapshot** — resolve such a finding by pointing at the commit that already fixed it, not by re-doing the work (PR #48 F5/F6 were generated against an older commit than the one that had already resolved them).
- When the author's revision improves on your suggestion, say so — credit the thinking, not just the diff.
- After a finding rewires an ordered flow or lifecycle, sweep docs, comments, and checklists for now-stale steps before the next round (batch them into one low-severity item).
- In verification tables, distinguish what you confirmed by reading code or running it from what you took on the author's word — otherwise the table reads as more thorough than it is.
- When recommending tests, include a short "not worth adding" list (already covered, framework behavior, trivial composition) — open-ended test asks invite sprawl.

## Retrospective

Every non-trivial PR ends with a brief retrospective — run it unprompted, don't wait to be asked. It is the **genuinely last step**: only after Codex's review is resolved _and_ the owner has approved, immediately before you execute the merge — not right after opening the PR, so it can capture lessons from the whole lifecycle including the review rounds. The fold is the **final pre-merge commit**, made once the owner says merge — **never committed during the implement/review rounds**; a lesson pushed mid-review is premature scope the reviewer will flag, so defer it and re-add it last (PR #45 T3: the retrospective fold was committed while resolving findings and had to be reverted, then re-added pre-merge). Capture what worked, what didn't, and what to change. Fold any lesson that generalizes into a durable convention **into this CLAUDE.md, in the same PR**, so the change and the lesson it produced live together. Keep it proportional: a typo fix has nothing to capture; a long multi-round PR may yield several. When a folded lesson is a *shared* convention, mirror it into the sibling guide (`AGENTS.md`) in the **same commit** — the two are a shared core plus a thin per-agent shell (title, attribution marker, footer, agent-specific sections). When reconciling them, `diff` and classify each hunk as shared-drift vs legitimate per-agent difference rather than overwriting (PR #39).

## Test fixtures from real captures

- A fixture built from real user data (a captured email, an API response, a log) carries PII beyond the obvious literals. Per-recipient **opaque tokens** — unsubscribe hashes, click-tracking IDs, profile UUIDs — identify the user even when no name or address is in sight, and the identity is often **encoded inside a token** (e.g. a tracking JWT whose base64 payload contains the recipient's email). Grepping for the literal address/name is not enough: redact every per-recipient token, then verify by also searching for **encoded forms** (the base64 of the address, etc.), not just the plaintext. Normalise to LF so golden lengths stay platform-stable. **Distinguish per-recipient from per-item tokens by *constancy across the artifact*:** a token identical everywhere in one capture identifies the *recipient* (redact it); one that *varies per item* (per job card, per row) identifies the *item*, not the user, and may stay (like a job title) — its value already rides every kept item. **And the no-leak test must guard *every* redacted key, not just the obviously-sensitive ones** — assert each holds its exact placeholder shape and **mutation-prove** it (revert one key to a realistic value; the test must flip red), or a future edit silently un-redacts a "minor" field while the scan stays green (PR #51: talent.com's `email_id`/`user_id` were constant across the email → per-recipient → redacted, while `pid`/`bpid` had 8/5 distinct values across the job cards → per-posting → kept; and the no-leak scan covered only 4 of the 9 redacted keys until Codex F1's mutation proof — un-redacting `t_id`/`c_id`/`d_sent` — forced widening it to all nine). **The no-leak test must not itself *embed* the real PII it guards:** `assert(!raw.includes('<real name/email>'))` re-commits that PII into the test source — assert the redaction **placeholder** is present and **extract-and-validate the whole occurrence set** per PII class (every email-shaped token is in the expected placeholder set; every greeting/name site equals the placeholder), mutating with **synthetic** values (`User`→`John`, placeholder→`john.doe@example.com`). And placeholder-**presence** alone is not enough — it stays green when a second real value is *added* beside the intact placeholder, so assert the exact set (count + value) and mutation-prove **both** an un-redaction *and* an additive insert (PR #52: Codex F-P2 caught the nexxt no-leak test embedding the real recipient name/email; re-review P2 then caught that the presence-only rewrite still passed an *added* second email — fixed by extracting every email/greeting occurrence and asserting the exact set).
- A fixture no test reads is a latent trap: it drifts unvalidated and gets silently adopted later (a pre-cleaned, truncated email once slipped in as a golden nobody had checked). Tie fixtures to tests with a **manifest/coverage check** (every fixture has a golden entry and vice versa), and before pinning a real capture as a golden confirm it is a faithful **raw** capture, not something already processed.
- **A real-capture fixture's fidelity gate is byte-identity, not length-equality against a stored/golden value.** The fixture is normalized (LF) and redacted; the stored value it's checked against may be a different line-ending era (CRLF) and still carry the real per-recipient tokens — so a raw length/byte compare spuriously fails. Gate on **byte-identity after normalizing both sides to LF** (CR-strip the stored value) **and masking the same per-recipient token regions in both**; keep redaction **exactly length-neutral**; and pin goldens **re-measured from the shipped LF fixture**, never copied from the stored value (kin to *"Numbers measured by a prototype are corridors, not goldens"*). (UK_DevOps 2026-06-19: a CRLF-era stored `CleanLength` of 7078 carried 20 in-text `\r` so an LF fixture could never match it on length, and the QP-decoded capture diverged only in `CLEAN_REGEX`-stripped `<head>` + redacted token regions — the LF clean was 7054; the length gate was impossible, the byte-identity gate held.)
- **A real capture's transport can diverge from the live ingestion path beyond CRLF/QP — normalize the transport-only differences in the gate, accept a faithful *prefix*, and let a *length-changing* divergence DEFER the fixture rather than ship a lossy one.** A capture pulled one way (an MCP `get_thread`) vs. what the live system ingested (the API the collector reads) can also differ in trailing inter-text **whitespace** — collapse runs before comparing. And an already-mapped sender's stored value may already carry a transform (its footer already cut at the *old* marker), so the gate accepts the stored value being a (masked, ws-collapsed) **prefix** of the full pipeline output, not only equality (the prefix *also confirms* the drift is live). But when the transport **QP-decodes** body bytes to C0 control chars (length-changing, un-committable) while the live path keeps QP **literal**, no faithful fixture is producible — **defer the mapping to the backlog**, don't commit a lossy capture. And the no-leak scan must enforce **every** PII class the redaction rule names — the recipient **name** in a greeting/`<title>`, not only the address + opaque tokens (UK_DevOps PR #42: cord/jooble/efc matched EQUAL after ws-collapse, the 3 drift senders matched as prefixes, nexxt deferred for QP control-chars, and Codex F2 caught a greeting name a token-only scan had missed).

## Working principles

- Prefer standard, built-in mechanisms over bespoke scripts unless there's a concrete reason.
- Prefer single-command forms with path flags (`git -C <dir>`, `npm --prefix <dir>`) over `cd <dir> && ...` chains — cleaner logs, error attribution, and allowlist matching.
- Treat generated files as a rendered contract, not noise: if a generated artifact and its source disagree, fix the **source** and regenerate; inspect the generated diff for unintended drift before committing.
- When you remove a check (a test, a guard, a CI filter), design its replacement in the same change — don't leave a silent gap.
- Conversely, when you add a file that a contract must react to — a CI trigger `paths:` filter, a lint/format glob, a manifest or coverage list — wire it into that contract in the same change. A new source-of-truth file the build consumes but no trigger watches is the same silent gap in mirror image: it can change and nothing re-runs. The same in reverse for a deploy **allow-list**: a tool that ships an explicit include-list (clasp's `.claspignore` un-ignoring each file by name, a bundler's entry list) **silently drops a new file that isn't added to it** — it deploys to nothing, no error — so a new script needs its own allow-list line, not merely to exist (PR #41: a new `.gs` beside `gmail-collector.gs` would never `clasp push` until `!vacancies-backup.gs` was added).
- **A runbook or doc step that prescribes editing a CI-*triggering* file inherits that workflow's execution — and its credentials.** "Edit `schema.json` and redeploy" is not just a local edit: the push fires the `paths:`-matched workflow, which runs under **its own** secret/token, so the procedure must satisfy *that* workflow's preconditions too, not only the local ones. Trace every file a manual step touches to the workflows that watch it before calling the step done (PR #41 T1→U1: the DR restore repoints `schema.json`'s base id, tripping `deploy-airtable.yml` → `apply-schema.js` under `AIRTABLE_SCHEMA_TOKEN`; granting the new base only to the local/data PAT still left the snapshot path's **merge** 403-ing in CI — the CI/schema PAT needed the same grant, in **both** restore cases).
- **A naming convention can be platform-load-bearing — don't apply it to a user-facing entry point by habit.** In Apps Script a **trailing-underscore** function is *private*: hidden from the editor's **Run** dropdown **and** the **Triggers** handler list. So a function the owner runs manually or assigns as a time-trigger handler must be **bare** (`backupVacancies`, `ensureDailyBackupTrigger`) even though every internal helper is `_`-suffixed — the same mark that makes helpers private makes an entry point unreachable (PR #41, owner-flagged: `ensureDailyBackupTrigger_` could not be selected in the Run dropdown).
- **Catch at the narrowest scope that can throw what you mean to handle.** A helper that both validates a precondition and does I/O (resolve `AIRTABLE_TOKEN`, then `UrlFetchApp.fetch`) wrapped in one `try/catch` masks the *config* failure as the *I/O* failure — a missing token became a synthetic "transient network error" that limped the run on instead of failing fast (PR #19 F-P2). Wrap only the call whose failure mode you're handling; let setup/programming errors propagate with their original stack. Prefer mapping a caught transport failure to a sentinel the caller classifies over catching everything and guessing the cause. **The masking can span two layers, and a new wrapper re-introduces it:** when a retry wrapper re-throws its transport failure *and* a caller catches-and-translates that throw, narrowing only the wrapper's `try` to the I/O call is *not enough* — the caller's broad catch still re-masks a post-fetch programming/classification error (e.g. `getResponseCode()` throwing) as the transient sentinel. Tag the wrapper's transport re-throw (e.g. `isAirtableTransportFailure`) so the translating caller maps *only* that failure mode to the sentinel and re-throws everything else; fix the catch scope at *every* layer that translates, not just the innermost (PR #20 F1 — the same F-P2 lesson, re-introduced when adding the retry wrapper).
- **A counter's increment unit must match the noun its message prints.** `N sub-batch upsert(s) failed`, incremented once per *record* during per-record isolation, reported "5 sub-batch" failures for a single 5-record sub-batch (PR #19 F-P3). When a tally feeds a human-readable summary, increment in the unit the message names, and pin the exact rendered string in a test so the granularity can't silently drift.
- **To make a side-effectful script testable, add a `--dry-run` that stops before the first side effect.** A tool that does git/network/process work is otherwise all-or-nothing to exercise; a dry-run that resolves and prints its plan, then exits (no branch, no push, no launch), gives an observable preflight a review can assert against — and doubles as a real safety feature. Prefer it to mocking the whole pipeline or skipping coverage, and verify the "dry" claim holds (e.g. assert `HEAD`/branch/branch-count unchanged) so it can't silently rot (PR #21: `dispatch-slice.sh --dry-run` resolves the branch and exits before any git/cloud op).
- **Exercising code that has a live trigger counterpart (hook, file-watcher, daemon, cron) can fire that live mechanism as a side effect of your test — isolate it.** Test inputs can trip the real trigger and mutate real state. Deploy the fix first, point the live mechanism elsewhere, or use inputs that can't match its filter — then check live state for contamination afterward. (PR #22: offline tests piped synthetic `gh pr create ... pull/N` payloads while the *old* buggy hook was still installed; it fired on the test commands and wrote a bogus `PR#42` into the real session title — the very bug under test, leaking into live state.)
- **A rollback/cleanup trap must be scoped to the window where undoing is still safe — disarm it the moment an external actor takes ownership.** A dispatcher that cuts a throwaway branch, pushes it, then hands it to a cloud session armed `trap rollback ERR` at branch creation and *disarmed it the instant the session launched*: past that point the launched session owns the branch, so a trivial later failure (the final `git checkout main`) must not roll back and delete work already in flight. And a rollback's reach must match where the resource was created — a branch made locally *and* pushed lives in two places — but auto-deleting the remote copy is unsafe when another actor may already hold it, so the rollback removes the *local* branch and **surfaces** the `git push origin --delete` command instead of running it (PR #24).
- **A fail-loud signal accumulated in code shared with a `DRY_RUN`/preview path must be classified for that path explicitly.** When a run has a preview mode that executes some stages for real but skips side effects, an alarm accumulator populated in a *shared* stage is silently dropped if the throw lives only in the non-preview branch. Decide per signal by its nature, not by the branch it happens to sit in: a **real-outage** signal whose underlying operation actually runs in preview (a Gmail read still happens under `DRY_RUN`) must **fire in preview too**, or preview validation silently misses the very failure it is meant to surface; a **side-effect-only** alarm (a write failure that can't occur with writes disabled; a template-drift alarm) stays **suppressed**. Hoist the canary out of the `if (dryRun) {…} else {…}` and gate each signal by its class. Generalises the F1 "don't swallow a co-occurring signal" lesson to the preview/real-run split (PR #25 F1[P2]: the new Gmail-read canary was incremented in the shared collect loop but thrown only in the real-run branch, so a persistent read outage under `DRY_RUN` logged then returned "complete").
- **A guard proven by concurrent peers decays to never-firing once the happy path removes the peers — and a single-invocation test won't catch it.** A cross-run cap gated on a same-run healthy sibling (`anyHealthy`) never fired for a persistently-stuck record: the success path `collected` the healthy siblings out of `CONFIG.QUERY`, so the stuck record was **solo** on every later run and the gate it depended on could never be satisfied again — exactly the population the cap existed to catch (PR #27 F1). Two habits. (1) When a gate depends on a peer/quorum/freshness signal, ask whether the *normal success path consumes that signal*; if so, **persist the proof once established and carry it forward (sticky)** — guard the first acquisition strictly, the continuation leniently. (2) A feature whose state persists **across invocations** (a Script Property, a DB row, a cursor) needs a test that **threads that state across several invocations** — every single-run test here passed while the counter silently froze across runs, the same false-assurance class as the stubbed-runtime and guard-vs-tested-predicate lessons above.
- **In a dual-path design (primary + fallback, old + new), a done-marker / dedup / idempotency invariant must hold in *both* paths — a fallback that performs only *part* of a multi-step marker corrupts dedup once the primary path resumes.** Post-M6.2 the authoritative done-marker is the RawEmails `Status` flip, but the first cut let the fallback keep the pre-2.0 "mark Gmail read" *alone*; an Airtable-outage fallback could mark Gmail read yet never flip `Status`, leaving rows the primary path re-screens on recovery (PR #29 F1). When a change makes one path's marker authoritative, **re-derive every *other* path's marker for the new contract — a fallback must not silently inherit the pre-change marker** — and prefer *removing* the second path to maintaining two half-markers (owner chose alert-and-stop: an Airtable outage alerts and stops rather than half-processing). Same family as the "don't swallow a co-occurring signal" / preview-vs-real-run lessons: enumerate every path a state change flows through and state each one's behaviour explicitly.
- **A matching/lookup key is not a writable field.** Schema-as-code that stores server-assigned ids for rename-safe matching (Airtable `fld…`/`tbl…` in `schema.json`) must **project those ids out of the create/write body** — the API returns an id in the *response* and rejects it in a *request*. Keep the id for matching; strip it via a dedicated `toCreatePayload` projection at *every* POST site (each individual add **and** the whole-table create path). Unit-test the projection on the **id-bearing missing-entity** case, because the happy path (everything already exists) never exercises the create path — a green suite won't catch the leak (PR #30 [P2]: the create path forwarded schema field objects verbatim, so a missing id-bearing field/table would have POSTed a stale `fld…` and broken the additive repair it advertises).
- **A lookup's *input* is not its *key*; emitting the input where a key is needed can be a silent no-op.** When you hand someone (or code) a value to **add as a key** to a first-match-wins, suffix/prefix-matched map, derive it at the map's **key granularity** — not the raw matched input. A more-specific input added as a new key is **appended after** the existing broader key, which still matches first, so the new entry never takes effect — and the bug is invisible because the value *looks* right. Read the matcher's real return value **and** iteration order (not the helper's *name*) before specifying what to output (PR #38 [P2]: the footer-freshness alert told the agent to emit `footerDomainOf_`'s full From host — `mail.uk.whatjobs.com` — as the `FOOTER_MARKERS` key, but the key is the registered domain `whatjobs.com`; for a drifted subdomain sender the host key would shadow-lose to the existing key in insertion order and the alert could never self-resolve. Fixed: emit the **matched key** for a drift, the **registered domain** for a new sender).
- **When you flag out-of-scope follow-up work, make it decision-ready — state its size and its correlation to the current change.** A follow-up surfaced as only *what it is* forces the owner to investigate scope before they can choose **fold into the current PR vs. spin off a separate session/PR**. Append a one-line **size** (rough effort — "one-line config change" vs "multi-file refactor + tests") and **relation to the current work** (does it touch the same files/contract; could it fold in cleanly?). A tiny, same-contract follow-up is a fold-in candidate — say so; a large or unrelated one is a clean spin-off. The decision stays the owner's; the assessment is what makes it informed (PR #31/#32: a one-line CI-paths fix that touched the very instructions test-trigger contract PR #31 was editing was spun off as its own session before its size/relatedness were stated — overhead the owner wouldn't have chosen with that information).
- **A transform that exists to strip a dangerous value must fail LOUD when its mechanism can't engage — never silently degrade to a path that ships the value while reporting success, even when the slice prompt *prescribes* that fallback.** A token-removing footer cut told to "fall back to a plain text cut when it can't find the leading `<a>`" recreates the exact per-recipient-token leak it exists to prevent *and* raises no alarm, so it reads as a clean `hit`. The safe behaviour is a `miss`/throw so the existing fail-loud alarm fires and a human fixes the cause. A spec-prescribed fallback is a hypothesis like any other — re-derive its safety before shipping it (this extends *a staged draft that over-promises ships the over-promise*, and *don't swallow a co-occurring signal* — both are silent-success-hides-a-real-failure) (PR #42 F1: Codex caught the link-mode text fallback; the fix fails closed — no anchor ⇒ `miss`). **Choosing `link` mode is itself a hypothesis to verify across *all* samples, not just the one you eyeballed:** `link` snaps the cut to the *nearest preceding `<a>`*, safe only if that anchor is a **stable footer element in every template**. A per-recipient token merely *leading* the marker is necessary but **not sufficient** — check what the nearest preceding `<a>` actually *is* per sample; if it is a footer action endpoint in one template but a **body content link** in another, `link` over-cuts the body there → prefer `text` (accepting a leading-token residue that already rides the kept body, the cord precedent). General form: a "snap to nearest X" heuristic mis-fires when X is a different kind of element in some input — verify the snap target's *identity* across the corpus before trusting it (PR #50: talent.com's nearest pre-marker `<a>` is the footer "Personalize my jobs" CTA in the digests but a body "More Jobs" link in the single-job template, so `link` was contraindicated).
- **Right-size the fix to the project's scale — a technically-correct review finding isn't always worth engineering away.** When a fix starts adding standing machinery to close a low-probability edge case, weigh it against the project's actual scale and risk tolerance: for a **solo, low-volume** project a negligible race / failure mode is often better handled as a **documented accepted risk** (an inline note + a revisit trigger + a cheap detection mitigation) than as a guard that adds permanent complexity. A reviewer can be *right that the risk exists* and it still not be worth fixing — surface the **cost/benefit and the accept-the-risk option** to the owner instead of reflexively building the heavier fix; the proportionality call is the owner's (PR #53: a Codex finding that a deploy retry's `sleep` lets a delayed older run overwrite a newer deploy was technically correct, but the `concurrency` + stale-SHA tip guard I built to close it was over-engineering for a ~10-PRs/day solo project — and the concurrency step even rested on an **unverified** claim that a GitHub Actions concurrency mutex guarantees newest-commit-last, which it does **not** [check the platform's docs before asserting such a guarantee]. Reverted to the simple retry, the race **recorded** as an accepted risk mitigated by the post-merge CI check + the CI-failure email).
